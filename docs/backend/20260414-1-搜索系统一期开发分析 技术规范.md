# 搜索系统一期开发分析

更新日期：2026-04-14

## 0. 文档定位与阅读方式

这份文档不是搜索系统的实施方案，也不是 UI 功能规划。

它要解决的问题只有一个：

> Lecquy 的搜索系统一期，到底应该以什么形态进入当前这套“runtime + 记忆 + RAG spike + 权限分层”的后端里？

因此全文的论证方式遵循三条规则：

- **区分三类信息**：`已知事实`（直接来自仓库现有代码与已定稿文档）、`基于现有代码的推断`（读了但没有显式写出的结论）、`建议方案`（文档给出的判断）
- **只围绕 Lecquy 当前阶段取舍**，不谈“未来哪天要做成 Perplexity”这种空话
- **参考项目只回答“该借哪一层”**，不复述项目的功能列表

阅读这份文档前建议先对齐四份已有文件：

- `docs/20260408-2-个人强 Agent 路线 开发规划.md`（路线定位）
- `docs/backend/20260408-7-Claude Code 能力借鉴路线 开发规划.md`（能力吸收顺序）
- `docs/backend/20260408-17-RAG Spike 边界 技术规范.md`（RAG 真实边界）
- `docs/backend/PaperQA 风格 RAG/20260408-4-PaperQA 风格 RAG 基线差距与实验建议 技术规范.md`（独立 RAG 基线）

## 1. 背景与目标

### 1.1 当前在总路线上的位置（已知事实）

`docs/20260408-2-个人强 Agent 路线 开发规划.md` 第 183-195 行明确写了优先级：

1. 记忆系统
2. Web Search
3. 自动任务
4. 未定义强化方向

并在同一文件 Phase 3（第 237-249 行）把搜索系统一期的重点列为：搜索后端接入、搜索结果归一化、搜索摘要与引用、网页抓取与抽取。

同时在 130 行写着非常关键的一句：

> 不要一开始把 Web Search 理解为一个 UI 功能，而要理解为 Agent 的基础设施能力。

也就是说，搜索系统在 Lecquy 总路线里的位置是：

- 在记忆系统一期之后
- 在自动任务之前
- 与 RAG spike 路线并列，但边界不同
- 属于“Agent 基础设施”，不是“前端功能模块”

### 1.2 为什么现在推进搜索是合理的（建议方案）

结合 `docs/README.md` 当前状态（第 17-30 行）可以看到：

- 记忆系统一期的 PG 底座、runtime dual-write、event extraction、retrieval injection、compact 原型、foresight 同步都已落地
- RAG spike 的最小表结构、chunk 策略、text-first 检索已具备
- 但都没有“接触外部世界”的能力，Agent 目前能记、能压缩、能召回，但不能获取当下信息

这对应 `docs/20260408-2-个人强 Agent 路线 开发规划.md` 第 99-101 行的判断：

> 记忆解决“过去我知道什么”，搜索解决“现在外部世界发生了什么”

因此 **现在做搜索，不是为了增加一个能力点，而是为了闭合“记忆 + 现实感知”这组最小组合**。如果不做，记忆系统再强也只是内部事件的归纳；如果做了，自动任务阶段才有资格谈“接触外部世界再决策”。

### 1.3 一期目标（建议方案）

把一期目标收敛为四条：

- 让 Agent 有一条稳定、可控、可观测的外部信息获取路径
- 让搜索结果以统一结构进入回答链路，而不是拼字符串
- 让搜索结果未来能自然沉淀进记忆系统与 RAG，但一期不直接耦合
- 让搜索能力以 tool / extension 形态接入，不污染 runtime 主链路

### 1.4 一期非目标（建议方案）

一期明确不做的事情，直接写清避免后面走偏：

- 不做 Perplexity 风格的一体化搜索页面
- 不做独立前端“搜索 UI”
- 不做多语言 reranker
- 不做搜索专用 embedding 服务
- 不做跨 Agent 协同搜索
- 不把搜索结果写入 `memory_items`
- 不把搜索结果直接写入 `knowledge_chunks`（原因见第 3 章）
- 不把 Docker / Docker Compose 作为开发路径（与 `AGENTS.md` 第 184 行一致）

## 2. 当前现状基线

基于对 `backend/src/` 的直接阅读：

| 能力 | 当前实现位置 | 状态 |

| Worker / Manager loop | `backend/src/agent/worker-runner.ts`、`manager-runner.ts` | 已具备分层角色 |

| 扩展工具装载 | `backend/src/extensions/index.ts` `createExtensionTools()` | 已具备；当前只挂载 `execute_sql` 与 `get_ai_archive_data` |

### 2.2 当前缺少的搜索相关能力（已知事实）

完整缺失的能力清单：

- 没有任何“外网 HTTP 出口”抽象，除 `bash` 工具以外，Agent 没有受控的 web 访问路径
- 没有“搜索工具”，`createSimpleTools()` 返回的工具清单（`read_file / bash / edit_file / write_file / skill / request_user_input / sessions_*` 加上 `createExtensionTools()`）没有任何一项面向外部搜索
- 没有搜索结果数据模型：`shared/src/session.ts` 里没有 `SearchResult / SearchHit / SearchCitation` 这类类型
- 没有 source card / evidence block / citation 渲染的 UI 与协议约定
- 没有 HTTP/网页抓取的结构化抽取能力（与 `bash` 自由命令不同，这是 agent 可控的调用）
- 没有搜索结果与 `runtime` / `memory_items` / `knowledge_chunks` 的连接点定义

### 2.3 可复用的现有模块（建议方案）

下面这些现有模块在搜索系统一期里应当**直接复用**，不再另起炉灶：

| 现有模块 | 复用方式 | 理由 |
| --- | --- | --- |
| `backend/src/extensions/index.ts` `createExtensionTools()` | 搜索工具从这里装载，失败不阻塞主流程 | 已验证的“工具加载但失败降级”路径，和 `execute_sql / get_ai_archive_data` 同类 |
| `backend/src/agent/tool-permission.ts` | 搜索工具默认 `auto`，抓取工具放 `preamble`，带副作用的登录/扣费工具放 `confirm` | 权限分档骨架已经存在，不需要为搜索重做一套 |
| `backend/src/agent/tools/index.ts` `createSimpleTools/createManagerTools/createWorkerTools` | 搜索工具通过该入口统一分发到三类角色 | 保持 manager/worker 边界一致 |
| `backend/src/core/prompts/prompt-layer-types.ts` 7 层 prompt | 搜索结果注入 `MemoryRecall` 层不合适，应走 `LiveTurn` 层或进入 assistant turn 的工具结果 | 不用新增层 |
| `@lecquy/shared` WS 协议 | 搜索进度与 citation 走现有 `tool_start / tool_end / message_delta`，不新增事件 | 保持协议稳定 |
| `session_events` 事件流 | 搜索调用记为 `session_tool_invoked / session_tool_finished` | 不需要新 entry type |
| `backend/src/db/*` 已有 PostgreSQL 基础 | 未来搜索缓存表直接落 PostgreSQL | 与 `AGENTS.md` 的“本机进程 + 本机 PostgreSQL”路径一致 |

### 2.4 当前 RAG 的真实边界（已知事实）

必须先把 RAG 边界讲清，否则搜索系统很容易被写成 RAG 的外皮。

`docs/backend/20260408-17-RAG Spike 边界 技术规范.md` 明确固定了：

- RAG 只做 spike，不进入主线交付（第 7-11 行）
- RAG 只解决“外部文档里有什么知识”，不解决“当下网络上发生了什么”（第 114-117 行）
- RAG 一期不接 runtime，不接 memory recall，不接 ws，不接前端（第 139-144 行）
- RAG 不做 embedding、不做 reranker、不做 citation 系统、不做多格式导入（第 30-34 行）

`docs/backend/PaperQA 风格 RAG/20260408-4-PaperQA 风格 RAG 基线差距与实验建议 技术规范.md` 第 170 行进一步写了：

> 所以现阶段 Lecquy 只有“骨架已通”，还没有“质量基线已立”。

这两件事合起来可以得出：**现有 RAG 不能直接充当搜索系统一期的后端**。理由有三：

1. RAG 以文档为输入，搜索以 query 为输入，两者生命周期完全不同
2. RAG 当前没有 ingestion pipeline，只能接受“已经准备好的纯文本 + metadata”
3. RAG 当前质量基线未立，把不确定的检索结果接进 Agent 主回答链路，风险比当前没有搜索更大

因此搜索系统一期需要自己的后端入口，不复用 `searchKnowledgeChunks()`；未来可以并列召回，但一期只冻结边界，不做融合。

## 3. 搜索系统的职责边界

### 3.1 三套系统各自解决什么问题（建议方案）

| 系统 | 解决的问题 | 输入 | 输出 | 生命周期 |
| --- | --- | --- | --- | --- |
| 搜索系统 | 当下外部世界发生了什么，以及用什么关键字能找到 | query、可选过滤条件 | 归一化的 `SearchHit[]` 与可选抓取结果 | 短期，per-turn |
| memory 系统 | 这个用户 / 这个会话过去发生过什么 | sessionId + userQuery | `MemoryRecallResult[]` | 长期，跨会话 |
| RAG 系统 | 我已经入库的外部文档里有什么 | query、可选 sourceFilter | `KnowledgeChunkHit[]` | 中期，与文档 ingest 绑定 |

### 3.2 未来自动任务如何依赖搜索（建议方案）

`docs/20260408-2-个人强 Agent 路线 开发规划.md` 第 148-153 行明确写了自动任务强依赖搜索。具体关系应该是：

- 自动任务触发 → 它调用搜索工具收集外部信息 → 结果进入任务 evidence
- 自动任务总结 → 任务完成后，有价值的搜索结果通过 consolidation 进入 `event / episodic` 记忆
- 自动任务复盘 → 失败原因如果来自外部世界变化，搜索结果作为证据

所以搜索对自动任务的贡献是：**提供 per-task 的外部证据**，而不是“自动任务执行搜索脚本”。

### 3.3 清晰边界表（建议方案）

为了避免后续代码和文档混，固定下面这张边界表：

| 维度 | 搜索系统 | memory 系统 | RAG 系统 |
| --- | --- | --- | --- |
| 主存储 | 可选搜索缓存表，不强制；结果默认不落库 | `memory_items`、`memory_jobs` | `knowledge_documents`、`knowledge_chunks` |
| 是否接 runtime 主链路 | 一期以 tool 形式接入，不直接改 runtime | 已通过 recall + compact 接入 | 一期不接入 |
| 是否接 ws | 借用现有 `tool_start / tool_end` | 借用 live turn 注入 | 不接入 |
| 结果消费者 | 当前 turn 的 LLM、未来 consolidation、未来自动任务 | 当前 turn 的 LLM、自动任务跟进 | 独立 spike，不消费主链路 |
| 允许写入的系统 | 写 session_events 工具调用记录；未来可写 consolidation 输入 | 写 memory_items | 写 knowledge_chunks（ingest 时） |
| 禁止的耦合 | 不直接写 `memory_items`、不直接写 `knowledge_chunks`、不绕过工具层 | 不把网页内容塞进 `memory_items` | 不代替 memory、不承接 search 输出 |

## 4. 参考项目分析

### 4.1 分析框架（建议方案）

对每个参考项目只回答四件事：

- 核心能力
- 适合借鉴的部分
- 不适合 Lecquy 当前阶段直接照搬的部分
- 匹配度判断（高 / 中 / 低）

这里的“匹配度”只评估在 Lecquy 当前阶段（runtime + memory 一期后 + RAG spike + 无外部出口）的契合度，不是绝对价值。

### 4.2 SearXNG

核心能力：聚合式元搜索引擎，把多个搜索引擎的结果归一化成统一 JSON 结构，支持本地部署与 HTTP API。

适合借鉴：

- 结果归一化的思路，特别是 `title / url / content / engine / score` 这套最小字段
- 本地优先的部署形态，与 Lecquy 的 local-first 原则一致
- 作为“搜索后端”的抽象，使得 Agent 不直接依赖任意具体引擎

不适合直接照搬：

- 它本身是一个完整 Web 服务，不该作为 Lecquy 进程内一部分
- 它默认需要独立部署，而 Lecquy 当前原则是“本机进程 + 本机 PostgreSQL 联调”，不以 Docker Compose 为主路径

匹配度：**高（作为外部搜索后端）/ 低（作为代码级依赖）**。

一期建议：**把 SearXNG 作为 HTTP 后端，Agent 端只暴露一个 `web_search` 工具消费它**。如果用户本机没有 SearXNG，默认禁用该工具并打印降级提示；不要默认启动一个 Compose。

### 4.3 Perplexica

核心能力：开源的 Perplexity 体验复刻，包含“搜索 + LLM 综合回答 + 引用”完整链路。

适合借鉴：

- “引用驱动回答”的思路，特别是 answer 必须和 source 绑定
- 搜索结果 → 摘要 → 引用 的三段式结构

不适合直接照搬：

- 它把“搜索前端”写成完整产品，Lecquy 一期不需要独立搜索页面
- 它把回答生成绑死在特定 prompt 模板里，而 Lecquy 的 prompt 层已经在 `prompt-layer-types.ts` 中规定了 7 层结构
- 直接把整个 Perplexica 拉进来，会引入一层和 runtime 冲突的 orchestration

匹配度：**中（作为“citation/evidence 结构”的参考）**。

一期建议：**只借鉴它的“答案 → 引用块 → 来源卡片”这三层数据结构**，不吸收它的 orchestration。

### 4.4 AgenticSeek

核心能力：本地 LLM + 隐私优先的自主搜索 Agent，强调离线与本地路由。

适合借鉴：

- 本地优先的原则与 Lecquy 路线高度一致
- 它把“搜索 + 浏览器 + 代码执行”组合成一个小型 agent 的做法，对自动任务阶段有参考价值

不适合直接照搬：

- 它是一个独立 agent runtime，和 `pi-agent-core` + `session-runtime-service.ts` 的双层 runtime 结构会打架
- 一期不应该为搜索引入第二个 agent loop

匹配度：**中（作为路线参考）/ 低（作为代码参考）**。

一期建议：**只吸收“本地优先、搜索作为 agent 能力而非 UI”这条设计原则**，不吸收其架构。

### 4.5 Crawl4AI + SearXNG

核心能力：SearXNG 负责检索、Crawl4AI 负责针对命中 URL 做结构化抓取与正文抽取，组合成“可输入给 LLM 的网页证据流”。

适合借鉴：

- **搜索与抓取分离**，Lecquy 一期应该继承这个分层
- Crawl4AI 提供的正文抽取 / 去广告 / 结构化标题结构，非常契合未来要做的 source card 与 evidence block
- 提供“网页证据”这件事独立成一个工具，不混入 LLM prompt

不适合直接照搬：

- Crawl4AI 对 embedding 与 RAG 支持较强，但 Lecquy 当前的 RAG spike 正要先稳住独立质量基线（见 `docs/backend/PaperQA 风格 RAG/20260408-4-PaperQA 风格 RAG 基线差距与实验建议 技术规范.md`），不应该在一期就把网页抓取输出直接塞进 `knowledge_chunks`
- Crawl4AI 默认配合 headless 浏览器，启动成本较高，不是一期默认形态

匹配度：**高（作为 Phase 3 分层参考）**。

一期建议：**明确搜索 vs 抓取的分层**；一期先做“搜索 + 最小 HTTP 抓取”，headless 浏览器推迟到 Phase 3 或更后。

### 4.6 OpenCLI

核心能力：针对特定网站的结构化抓取工具链，不是全网搜索。

适合借鉴：

- “网站适配器”思路，这类能力未来作为 extension tool 挂到 `backend/src/extensions/`
- 对特定来源（比如某个文档站）做高质量结构化抽取

不适合直接照搬：

- 它不是搜索系统，不能替代一期的 `web_search`
- 它的形态更适合作为后续 skill / extension 的具体案例

匹配度：**低（一期）/ 中（Phase 3 之后作为特定数据源适配）**。

一期建议：**一期不考虑 OpenCLI；它在 Phase 4 之后再以 extension 形式加入**。

### 4.7 PaperQA / PaperQA2（只从“搜索 / evidence / retrieval”视角）

核心能力（仅取搜索/证据相关）：

- 从 `docs/backend/PaperQA 风格 RAG/20260408-4-PaperQA 风格 RAG 基线差距与实验建议 技术规范.md` 第 198-207 行可以直接看到，PaperQA2 的核心能力是：`retrieval → evidence selection → answer synthesis` 三段式，而不是“一次 topK 完事”
- `Context` 对象 + `score` + `cutoff` 把证据打成结构化对象，而不是字符串

适合借鉴：

- “证据对象化”这一结论，对搜索系统完全适用：搜索结果不能只以字符串进入 prompt，必须是结构化 hit + score + 可追溯来源
- 两段式（候选池 → 精筛证据）的思路，未来也适用于 `web_search + web_summarize` 两段式搜索

不适合直接照搬：

- PaperQA2 是文档 QA 系统，不是 web search 系统，它不解决“当下世界发生了什么”
- 一期搜索不需要 per-chunk LLM relevance score，那是 RAG 路线里的 evidence selection 层

匹配度：**中（作为“结构化证据”设计思路）**。

一期建议：**借它“证据是对象，不是字符串”这一条**；不要照抄它的 per-chunk score 流水线到搜索系统。

### 4.8 小结

- 一期最值得作为**基础后端**的：SearXNG
- 一期最值得作为**分层范式**的：Crawl4AI + SearXNG（搜索与抓取分离）
- 一期最值得作为**数据结构**参考的：Perplexica 的 source card、PaperQA 的 `Context` 对象化
- 一期**不建议吸收为架构**的：Perplexica 全链、AgenticSeek runtime、Crawl4AI 完整栈

## 5. Lecquy 搜索系统一期建议形态

### 5.1 一期最小目标（建议方案）

**让 Agent 拥有一个稳定、可控、可观测的外部搜索能力**，以工具形式接入，能在当前 turn 被调用、在 session_events 中留痕、不污染 runtime 主链路，不混入 memory/RAG。

换一种说法：完成一期后，用户在 simple 模式下问“查一下今天 xxx 新闻”，Agent 应该能调 `web_search`、拿到归一化结果、把结果写进当前回答并携带引用。

### 5.2 建议分层（建议方案）

参考 `AGENTS.md` 里 `Claude Code` / `Codex` 风格，把搜索系统自顶向下分成五层：

1. **后端抽象层**：`SearchBackend` 接口，屏蔽具体引擎；一期只实现 `SearxngBackend`
2. **工具层**：`web_search` 工具；可选 `web_fetch`；走 `createExtensionTools()` 注入
3. **归一化层**：把 backend 返回的结构归一化为 `SearchHit` 与 `SearchCitation`
4. **回答注入层**：以工具结果的形式进入 agent loop，不新增 prompt 层
5. **可观测层**：复用现有 `tool_start / tool_end`，加上必要的 `detail` 字段

### 5.3 推荐技术路线（建议方案）

- 搜索后端：**SearXNG（本机部署）**，通过 HTTP JSON API 调用
- 搜索工具：TypeScript 原生 `fetch`（Node 18+ 已内置），不引入重型 HTTP 库
- 归一化：在 `backend/src/search/` 下写 `types.ts / normalizer.ts / searxng-backend.ts / web-search-tool.ts`
- 可选抓取：一期只做 **基于 `fetch` 的最小 HTML 正文抽取**；headless 浏览器推迟
- 持久化：**一期不建缓存表**；如果未来需要，再在 PostgreSQL 新建 `search_queries` 与 `search_hits`
- 权限：`web_search` 放 `auto`（只读，风险低）；`web_fetch` 放 `preamble`（输出不确定，需 agent 先说明）；带登录 / 带 cookie 的抓取放 `confirm`

### 5.4 推荐先后顺序（建议方案）

- 先做后端抽象 + SearXNG adapter + 归一化
- 再把 `web_search` 注入 extension tools，先只在 simple 模式可用
- 再做最小引用 / source card 约定，让回答能指向结果编号
- 再做 `web_fetch` 的最小抓取
- 最后再谈与 consolidation、RAG、自动任务的联动

### 5.5 一期非目标（建议方案）

重复强调，避免走偏：

- 不做独立搜索 UI 页面
- 不做搜索结果的独立前端列表（仅在 chat 消息里呈现 source card）
- 不做 headless browser
- 不做跨引擎 rerank 与 learning-to-rank
- 不做多用户共享搜索缓存
- 不建 embedding 层
- 不把抓取结果写 `knowledge_chunks`

## 6. 后端架构建议

### 6.1 搜索请求从哪里发起（建议方案）

只允许两条发起路径：

- **Agent 工具调用**（主路径）：LLM 决定调用 `web_search`，经 `backend/src/agent/tools/index.ts → createExtensionTools()` 装载的工具入口触达搜索后端
- **未来自动任务内部调用**（未实现）：心跳任务系统未来通过同一 tool contract 调用，不旁路

明确禁止的路径：

- 不在 `backend/src/runtime/session-runtime-service.ts` 的主 loop 里硬编码搜索调用
- 不从 WebSocket handler 直接调搜索，绕过 agent loop
- 不从前端直接访问搜索后端

### 6.2 搜索结果如何标准化（建议方案）

一期冻结两层结构：

- `SearchHit`：原始搜索后端结果的归一化形式
- `SearchCitation`：进入回答链路时的引用形式

归一化由 `backend/src/search/normalizer.ts` 负责，工具 `execute()` 里只返回 `SearchHit[]` 与最小元信息；`SearchCitation` 只在需要进入回答上下文时由工具 wrapper 派生。

### 6.3 搜索结果如何进入回答链路（建议方案）

一期**不新增 prompt 层**。搜索结果通过两条已经存在的通道进入 LLM：

- 工具结果：`web_search` 的 `execute()` 返回的 `AgentToolResult`，`content` 是结构化文本块，`details` 保存原始 hits（不进 LLM）
- assistant turn 的回答：让 LLM 消费工具结果后，在回答里自然引用

这与现有 7 层 prompt（`PromptLayer.System` ~ `PromptLayer.LiveTurn`）完全兼容；搜索结果本质是 `LiveTurn` 内的工具结果，不需要放到 `MemoryRecall` 层。

### 6.4 是否要先做 citation / source card / evidence block（建议方案）

分拆成三个问题回答：

- **citation**：一期需要做最小版，因为没有引用就无法区分“回答里的事实”和“LLM 的话”。最小版只要求：工具返回里每条 hit 有稳定 `hitId`，回答里允许用 `[n]` 索引该 hit
- **source card**：一期**不做独立前端组件**；可以先在 chat 消息的 tool 结果块里以文本方式渲染 title + url，前端复用现有 `MessageItem` 路径
- **evidence block**：一期**不做**，它和 PaperQA 风格的 `Context.score` 绑定，是 RAG 路线的范畴

### 6.5 是否先接成 tool 而不是直接硬接 runtime 核心（建议方案）

**必须先接成 tool**。原因：

- `docs/backend/20260408-7-Claude Code 能力借鉴路线 开发规划.md` 第 138-149 行已经提出“内部扩展点”优先，不在主链路堆逻辑
- `backend/src/extensions/index.ts` 已经为类似能力提供了注入口（当前挂 `execute_sql` 与 `get_ai_archive_data`）
- 工具形态天然复用：`tool-permission.ts` 的三档权限、`session_events` 的 `session_tool_invoked / session_tool_finished`、WS 的 `tool_start / tool_end`
- 工具形态天然可禁用：未连 SearXNG 时，`createExtensionTools()` 的 `try/catch` 会跳过这个工具而不阻塞主流程

### 6.6 与权限、mode、worker/manager 的关系（建议方案）

- **权限**：`web_search` 默认 `auto`；`web_fetch` 默认 `preamble`；未来带登录的抓取默认 `confirm`
- **mode**：simple 和 plan 模式都允许；这与现有 `createSimpleTools()` 行为一致
- **manager**：不加入 `MANAGER_WHITELIST`，manager 只做编排，不直接搜索
- **worker**：加入 worker 允许列表，复杂搜索动作应该在 worker 内发生
- 这样的划分和 `backend/src/agent/tool-permission.ts` 现有结构兼容，不需要引入第四类角色

### 6.7 与 memory、RAG、自动任务的连接点（建议方案）

**一期只冻结接口，不实现融合**：

- 与 memory 的连接点：搜索结果**不直接写 memory**；只有当 Agent 在回答里确实引用，并通过未来的 consolidation job（见 `docs/backend/20260408-7-Claude Code 能力借鉴路线 开发规划.md` 的 autoDream 思路）识别为“值得长期记忆”时，才以 `event` 形式写入 `memory_items`
- 与 RAG 的连接点：搜索结果**不直接写 `knowledge_chunks`**；未来若用户主动说“把这个页面存下来”，再单独走 `ingestKnowledgeDocument()`
- 与自动任务的连接点：未来心跳任务通过同一个 tool contract 调用搜索，不旁路；这保证了一期做对的抽象以后不用重写

## 7. 数据模型与接口草案

### 7.1 建议的数据流（建议方案）

```
LLM 决策
   │
   ▼
web_search(query, options)        ← agent tool
   │
   ▼
SearchBackend.search(query)       ← 抽象层
   │
   ▼
SearxngBackend.search(query)      ← 一期实现
   │
   ▼
SearchHit[] + 原始 payload        ← 归一化
   │
   ▼
AgentToolResult                   ← 返回给 agent loop
   │
   ▼
LLM 消费（可能后续触发 web_fetch 或直接回答）
   │
   ▼
session_tool_finished event       ← 留痕
```

### 7.2 建议的 TypeScript 接口草案（建议方案）

只写分析级草案，不做实现爆炸。文件位置建议在 `backend/src/search/types.ts`：

```ts
// 搜索引擎可观测标识
export type SearchEngineId = 'searxng' | 'stub' | (string & {})

// 标准化后的搜索结果 hit
export interface SearchHit {
  readonly hitId: string          // 稳定短 ID，供 LLM 在回答里引用
  readonly url: string
  readonly title: string
  readonly snippet: string        // 引擎返回的短描述
  readonly engineId: SearchEngineId
  readonly rankInEngine: number   // 该引擎内原始排名
  readonly score?: number         // 可选合并分数，一期可不填
  readonly language?: string
  readonly publishedAt?: string   // 引擎声明的发布时间，可选
  readonly metadata?: Record<string, unknown>
}

// 进入回答链路时的 citation
export interface SearchCitation {
  readonly hitId: string
  readonly url: string
  readonly title: string
}

// 一期搜索入参
export interface WebSearchInput {
  readonly query: string
  readonly topK?: number
  readonly language?: string
  readonly safeSearch?: 'off' | 'moderate' | 'strict'
  readonly engines?: SearchEngineId[]
}

// 一期搜索出参
export interface WebSearchOutput {
  readonly query: string
  readonly hits: SearchHit[]
  readonly totalCandidates: number
  readonly degraded: boolean       // true 表示后端不可用，已降级
  readonly degradedReason?: string
}

// 抓取一期入参（可选，Phase 3 才上）
export interface WebFetchInput {
  readonly url: string
  readonly maxChars?: number
}

export interface WebFetchOutput {
  readonly url: string
  readonly finalUrl: string
  readonly status: number
  readonly contentType?: string
  readonly title?: string
  readonly mainText?: string       // 一期只做最小 HTML 正文抽取
  readonly truncated: boolean
}

// 后端抽象
export interface SearchBackend {
  readonly id: SearchEngineId
  readonly isAvailable: () => Promise<boolean>
  readonly search: (input: WebSearchInput) => Promise<WebSearchOutput>
}
```

### 7.3 最小后端模块划分建议（建议方案）

```
backend/src/search/
  types.ts                 // 上述类型
  searxng-backend.ts       // SearxngBackend 实现
  stub-backend.ts          // 未连 SearXNG 时的兜底，固定返回 degraded=true
  normalizer.ts            // 原始 payload → SearchHit
  web-search-tool.ts       // AgentTool<WebSearchInput>
  web-fetch-tool.ts        // Phase 3 再加
  index.ts                 // 对外导出
```

并在 `backend/src/extensions/index.ts` 的 `createExtensionTools()` 里新增一段（示意）：

```ts
try {
  tools.push(createWebSearchTool())
} catch (error) {
  logger.warn('扩展工具 web_search 加载失败:', error)
}
```

这样搜索工具就和 `execute_sql / get_ai_archive_data` 一样走“失败降级不阻塞”路径。

### 7.4 搜索结果统一结构原则（建议方案）

- hits 之间**只靠 `hitId` 做引用**，不依赖原始 URL（URL 可能过长或含敏感参数）
- `metadata` 是黑盒，只用于调试与后续 consolidation 分析，不进入 prompt
- `degraded` 必须显式存在，LLM 能看到搜索降级并在回答里如实说明
- 任何后端新增时，先增加 `SearchEngineId` 取值，不破坏现有结构

## 8. 分阶段开发顺序

每个阶段都写清：**目标 / 范围 / 依赖 / 验收标准**。阶段之间严格保证“上一阶段落地 + 验收通过，才开启下一阶段”。

### 8.1 Phase 1：最小可用搜索后端

目标：让 Agent 在 simple 模式下可以成功调用 `web_search` 并拿到归一化结果。

范围：

- `backend/src/search/{types.ts, searxng-backend.ts, stub-backend.ts, normalizer.ts, web-search-tool.ts, index.ts}`
- 在 `createExtensionTools()` 注入 `web_search`
- 在 `AUTO_TOOLS` 加入 `web_search`
- 纯后端实现，不动前端

依赖：

- 本机已跑通一个 SearXNG 实例（允许开发机另行启动，但不作为 Lecquy 默认依赖）
- 环境变量 `SEARXNG_BASE_URL`（默认空，为空时使用 `stub-backend.ts`，并返回 `degraded=true`）

验收标准：

- 单测覆盖 `normalizer.ts`，给 fixture SearXNG JSON，输出稳定 `SearchHit[]`
- 后端未配置时 tool 调用返回 `degraded=true` 且不抛异常
- 后端已配置时，手工用 `simple` 模式发送“查一下 TypeScript 5.9 是什么时候发布的”，能看到 WS `tool_start / tool_end` 和一段带 hits 的工具结果
- `session_events` 中出现对应的 `session_tool_invoked / session_tool_finished`

### 8.2 Phase 2：搜索结果摘要与引用

目标：让 Agent 的回答里能自然出现引用编号，并把 hits 以 source card 的最小形式展示给用户。

范围：

- 在 system prompt 的工具使用说明里增补：鼓励在回答里使用 `[n]` 引用某个 hit
- 实现一个 `SearchCitation` 导出，供后续 consolidation 使用
- 前端仅在 `MessageItem` 的 tool 结果块里加最小的 title / url 渲染（不新增组件，不新增 WS 事件）

依赖：

- Phase 1 完成
- `docs/frontend/20260417-1-Markdown 渲染排障 技术规范.md` 已经稳定，`[n]` 不会被 markdown 吃掉

验收标准：

- 同样的查询下，回答里出现 `[1]` `[2]` 等编号
- 前端展示的 hit 项能点击跳转
- 工具结果块能与 assistant 文本块**在视觉上区分**（复用现有组件，不重做）

### 8.3 Phase 3：网页抓取与结构化抽取

目标：让 Agent 在看过搜索结果后，能调用 `web_fetch` 取到可读的正文片段。

范围：

- `backend/src/search/web-fetch-tool.ts` + 最小 HTML 正文抽取
- `web_fetch` 默认 `preamble` 档
- 明确抓取大小上限（如 50KB 正文、总耗时 ≤ 5s）
- 降级策略：超时 / 非 HTML / 巨大页面，返回 `truncated=true` 与解释

依赖：

- Phase 2 完成
- 已决定不引入 headless browser；若某条需求必须 JS 渲染，本阶段明确打回

验收标准：

- 能抓取常见 UTF-8 HTML 页面，提取 `<title>` 与正文
- 抓取失败时 LLM 能看到清晰 error 原因
- 工具调用在 `session_events` 留痕

### 8.4 Phase 4：与知识沉淀 / RAG / 自动任务联动

目标：把搜索能力真正接入长期系统。

范围（按优先级从高到低，本阶段可能只做前两项）：

- 搜索 + 抓取结果作为 **consolidation 输入**，由 autoDream 风格 job 选择性写入 `event` 记忆
- 搜索结果作为 **自动任务 evidence**，通过未来心跳任务系统调用同一工具
- **用户主动保存**页面到 RAG，调用 `ingestKnowledgeDocument()`
- 并列召回：memory recall → knowledge recall → search results 三路并存

依赖：

- `docs/backend/20260408-7-Claude Code 能力借鉴路线 开发规划.md` 中 autoDream 层已启动
- RAG 路线已脱离 spike，完成 `docs/backend/PaperQA 风格 RAG/20260408-4-...` 中第 5 节的三项实验
- 心跳任务系统已经落地

验收标准：

- 搜索 consolidation 能稳定跑通，并且没有把随手结果污染 memory
- 用户保存页面时 RAG 能正常 ingest
- 自动任务内部使用 `web_search` 时和 simple 模式行为一致，不需要特殊分支

## 9. 风险与决策建议

### 9.1 技术风险（建议方案）

- **SearXNG 本机可用性**：开发者机器可能没有 SearXNG；解决：`stub-backend.ts` 兜底，`degraded=true`，不阻塞一期联调
- **Node 原生 `fetch` 稳定性**：Node 18+ 已稳定；如果 pi-agent-core 环境里仍出现 undici 问题，退回到 `node-fetch` 前要先对比体积
- **HTML 抓取的编码与解析**：常见坑在于 GBK / Big5；一期只保证 UTF-8，其余明确报 `degraded`
- **抓取导致 worker 阻塞**：`web_fetch` 必须强制 timeout；与现有 `bash` 工具的 `120_000 ms` 相比，抓取建议 5-10s 硬超时

### 9.2 产品风险（建议方案）

- **回答把模型幻觉包装成引用**：解决：一期就让工具结果在 `AgentToolResult.content` 里把每条 hit 的 `hitId` + title + url 明确给 LLM；回答里要求使用 `[n]` 指向某个确切 hitId
- **用户以为搜索永远在线**：解决：把 `degraded` 字段当成一等公民，工具结果块里显式告诉用户当前没有可用搜索后端
- **搜索被当成独立产品**：解决：一期禁止前端做独立搜索页面，保持搜索出现在对话流中

### 9.3 复杂度风险（建议方案）

- **过度抽象**：一期只做一个 `SearxngBackend`，不要先做 multi-backend 合并策略；直到出现第二个 backend 才考虑 merge 策略
- **过早融合**：不要把搜索结果同时写 memory / RAG / session_events 三份，未来只会制造不一致
- **过早做 rerank**：`docs/backend/PaperQA 风格 RAG/20260408-4-...` 已经说明 recall 是第一瓶颈，不是 rerank；搜索侧同理

### 9.4 容易走偏的点（建议方案）

- 把 `web_search` 直接接进 runtime 主循环（禁止）
- 把搜索结果塞进 `memory_items` 去蹭 recall（禁止）
- 把 `knowledge_chunks` 当搜索缓存（禁止）
- 为了看起来像 Perplexity，提前做前端搜索页面（推迟）
- 为了看起来像 PaperQA，对每个 hit 跑 LLM relevance score（推迟）
- 把 SearXNG 作为 Lecquy 自带依赖强制拉起（禁止，违反 `AGENTS.md` 第 184 行开发期原则）

### 9.5 最终建议路线（建议方案）

一句话：

> 搜索系统一期以 **工具形态 + SearXNG 后端 + 归一化结构** 落地；先闭合 simple 模式下“查-答-引用”的最小链路；抓取与联动在后续阶段再进。

之所以选这条路线，不是因为它最炫，而是因为它满足三个硬约束：

- 不改 runtime 核心，不破坏当前 7 层 prompt 与权限三档
- 不污染 memory 与 RAG，两个系统边界仍然干净
- 不默认引入 Docker / headless 浏览器，和当前“本机进程 + 本机 PostgreSQL”开发期原则一致

推进时严格守住阶段顺序：Phase 1 是地基，Phase 2 是最小可交互版，Phase 3 是抓取能力，Phase 4 才谈长期沉淀。中间跳步会让“搜索”再次退化成“UI 功能”，而不是 Agent 基础设施。
