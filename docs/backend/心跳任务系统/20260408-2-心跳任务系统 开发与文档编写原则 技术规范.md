# 心跳任务系统开发与文档编写原则

更新日期：2026-04-08

## 1. 开发原则

### 1.1 先设计状态机，再写调度代码

这条路线禁止先写一堆定时任务脚本，再回头解释状态。

先后顺序必须是：

1. 状态机
2. 触发器
3. 执行器
4. 恢复机制

### 1.2 主动任务必须可解释

每个心跳任务都要能回答：

- 为什么被触发
- 依据什么状态触发
- 写回了什么结果
- 如何避免重复执行

### 1.3 优先利用现有底座

优先复用：

- runtime 事件流
- todo / foresight
- PG 存储
- memory recall

不要在早期重新造第二套状态持久化系统。

## 2. 文档编写原则

每次新增文档至少包含：

1. 任务类型
2. 触发条件
3. 状态流转
4. 去重策略
5. 恢复策略
6. 风险

## 3. 统一词汇

在本路线文档中，尽量统一使用：

- `心跳任务`
- `触发器`
- `状态机`
- `去重`
- `恢复`
- `主动任务`

## 4. 变更边界

允许改：

- 与主动任务直接相关的新目录
- runtime 中有限的触发点
- PG 表与状态持久化

默认不改：

- 前端 UI
- RAG 主链路
- 无关的 memory schema
