# 心跳任务系统开发规划

更新日期：2026-04-08

## 1. 目标

这条路线的目标是模仿 OpenClaw 和 Claude Code 的“心跳任务 / 主动触发任务”设计，让系统具备在适当时机主动推进任务的能力。

目标能力包括：

- 定时或事件驱动触发
- 待办检查
- 任务去重
- 任务恢复
- 长期任务推进

## 2. 当前基线

当前仓库已经具备一些基础：

- `todo_updated` 事件
- `foresight` 写入链路
- runtime 会话事件流
- compact / memory / PG 持久化底座

当前缺少：

- 独立的心跳调度器
- 主动任务触发条件
- 心跳任务状态机
- 去重与恢复策略

## 3. 参考项目

主参考对象：

- OpenClaw
- Claude Code

重点关注：

- 心跳触发时机
- 任务状态流
- 任务恢复与中断
- 主动任务如何使用已有上下文和记忆

## 4. 分阶段开发顺序

### Phase 1：设计状态机

明确：

- 心跳任务的生命周期
- 任务状态
- 去重规则

### Phase 2：设计触发器

明确：

- 定时触发
- 事件触发
- 用户空闲或会话结束后的触发

### Phase 3：最小执行器

实现：

- 最小 heartbeat runner
- 最小任务持久化
- 最小恢复机制

### Phase 4：接入记忆与上下文

明确：

- 心跳任务如何读取 memory / foresight
- 心跳任务如何写回状态

## 5. 验收标准

一期完成至少需要：

1. 有明确的心跳任务状态机
2. 有最小调度与去重机制
3. 至少一类主动任务可以执行并持久化状态
4. 文档能清楚说明与 OpenClaw / Claude Code 的差异

## 6. 当前不做

- 不直接做前端 UI
- 不引入分布式调度
- 不把这一条路线和 RAG 混做
