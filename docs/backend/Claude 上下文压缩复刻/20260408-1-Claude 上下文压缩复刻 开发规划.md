# Claude 上下文压缩复刻开发规划

更新日期：2026-04-08

## 1. 目标

这条路线的目标是尽量高保真复刻 Claude Code 的 TypeScript 侧上下文工程：

- context builder 组织方式
- compact 触发与历史裁剪策略
- memory block / compact block 的固定模板
- cache-friendly 的上下文拼接顺序
- simple / plan / final-answer 等路径的统一上下文构建纪律

不承诺的部分：

- Anthropic 服务端 prefix cache 的实际实现
- 隐藏 system prompt
- 模型服务端的专有优化

## 2. 当前基线

当前仓库已经具备这条路线的基础：

- 已有 unified builder：`backend/src/runtime/context/augmented-context-builder.ts`
- 已有 memory recall 模板：`backend/src/runtime/context/templates/memory-recall.template.ts`
- 已有 compact summary 模板：`backend/src/runtime/context/templates/compact-summary.template.ts`
- `simple` 和 `plan final answer` 已接统一 builder
- compaction 识别已从字符串匹配收口为结构化判断

当前仍未完成的部分：

- 还没有系统性对照 `Kuberwastaken-src` 的完整上下文工程
- 还没有把 simple / plan / retry / resume 的全部上下文构建路径完全统一审计
- 还没有形成一套“Claude 对标差异清单”

## 4. 分阶段开发顺序

### Phase 1：对标差异审计

输出目标：

- Claude Code 的上下文拼接链路图
- Lecquy 当前上下文拼接链路图
- 差异清单：哪些已对齐，哪些未对齐，哪些不该照搬

### Phase 2：上下文路径补齐

重点：

- simple / plan / final-answer / retry / resume 是否共用 builder
- 哪些路径仍然手工拼接 contextMessages
- memory recall / compact summary / recent tail 的顺序是否完全固定

### Phase 3：compact 行为复刻

重点：

- compact 触发条件是否更接近 Claude Code
- recent tail 的选择是否更接近 Claude Code
- compaction event 的载荷结构是否需要调整

### Phase 4：cache-friendly 收口

重点：

- prefix 稳定区块
- memory / compact 模板稳定性
- 各路径 prompt 装配顺序一致性

## 5. 验收标准

达到下面标准，才算这条路线的一期完成：

1. Lecquy 能列出自身与 Claude Code 上下文工程的差异清单
2. runtime 侧所有主要路径都走统一的上下文构建入口
3. memory recall / compact summary 模板只定义一次
4. compact 行为和上下文顺序有明确、稳定、可测试的约束
5. 文档中能清楚说明“哪些部分已经对齐 Claude，哪些还没有”

## 6. 暂不做的事

- 不在这一条路线里扩 RAG 功能
- 不在这一条路线里新增 profile / episodic 新能力
- 不在这一条路线里做前端 UI
- 不在这一条路线里处理 Anthropic 服务端级缓存实现
