# Claude 上下文压缩复刻开发与文档编写原则

更新日期：2026-04-08

## 1. 适用范围

这份文档约束本路线下的两类输出：

- 代码开发
- 新增文档编写

适用目录：

- `docs/backend/Claude 上下文压缩复刻/`
- `backend/src/runtime/context/`
- `backend/src/memory/`
- 与上下文构建直接相关的 runtime 路径

## 2. 开发原则

### 2.1 先做对标，再做改动

任何“复刻 Claude Code”相关改动，都必须先说明对应参考文件和目标行为。

禁止：

- 只凭印象改上下文顺序
- 不说明参考来源就重构 builder

### 2.2 优先统一路径，不优先扩功能

这条路线的首要任务是统一上下文工程，不是增加新的记忆能力。

优先级：

1. 统一 builder
2. 固定模板
3. 固定顺序
4. 补齐路径
5. 再谈效果优化

### 2.3 不破坏现有主链路

任何改动必须保证：

- `simple` 继续可用
- `plan` 继续可用
- `final-answer` 继续可用
- PG 和 memory 写入链路不回退

## 3. 文档编写原则

### 3.1 每次新增文档都要写清 4 件事

1. 当前基线是什么
2. 参考项目是什么
3. 本轮要改什么
4. 明确不改什么

### 3.2 记录“对标差异”，不要只写实现细节

这条路线的核心不是“写了哪些代码”，而是“与 Claude Code 还差什么”。

所以每次新增文档，至少包含一节：

- 已对齐点
- 未对齐点
- 有意不对齐点

### 3.3 统一词汇

在本路线文档中，尽量统一使用这些词：

- `上下文工程`
- `统一 builder`
- `memory recall block`
- `compact summary block`
- `recent tail`
- `cache-friendly`
- `共享前缀`

## 4. 变更边界

本路线文档和代码允许改：

- 上下文 builder
- context template
- compact 行为
- contextMessages 顺序
- runtime 侧的上下文装配调用点

本路线默认不改：

- RAG 设计
- 前端 UI
- 数据库 schema 的大重构
- WebSocket 协议

## 5. 每轮文档输出模板

建议后续新文档都包含以下章节：

1. 目标
2. 当前基线
3. Claude Code 对标点
4. 本轮改动范围
5. 不改范围
6. 验收标准
7. 风险与未决项
