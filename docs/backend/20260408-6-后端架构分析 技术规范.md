# Lecquy 后端架构分析

更新日期：2026-04-01

## 1. 主结论

当前后端的正确主链路已经不是过去文档里常写的 `SessionService(session-v2)`，而是：

`WebSocket 入口 -> simple/plan handler -> Agent runner -> SessionRuntimeService(runtime) -> shared 协议`

也就是说：

- WebSocket 仍然是主交互入口
- `simple / plan` 仍然是当前最稳定的模式边界
- **runtime** 已经成为真正的会话中枢
- `session-v2` 现在更多是残余类型和历史实现，不再是主持久化链路

## 2. 当前主链路

核心文件：

- [`backend/src/server.ts`](../../backend/src/server.ts)
- [`backend/src/ws/chat-ws.ts`](../../backend/src/ws/chat-ws.ts)
- [`backend/src/ws/simple-handler.ts`](../../backend/src/ws/simple-handler.ts)
- [`backend/src/ws/plan-handler.ts`](../../backend/src/ws/plan-handler.ts)
- [`backend/src/runtime/session-runtime-service.ts`](../../backend/src/runtime/session-runtime-service.ts)
- [`backend/src/runtime/pi-session-core/session-manager.ts`](../../backend/src/runtime/pi-session-core/session-manager.ts)

主链路的职责分工：

- `chat-ws.ts`
  - 连接生命周期
  - 协议事件收发
  - 模式分发
- `simple-handler.ts / plan-handler.ts`
  - 模式级请求编排
- `agent-runner / manager-runner / worker-runner`
  - 具体 Agent 调用与工具执行
- `SessionRuntimeService`
  - 会话解析
  - projection 刷新
  - todo / pause / notify / history
- `shared`
  - 前后端协议单一事实源

## 3. runtime 为什么是当前中枢

runtime 的关键点不是“会写文件”，而是它把 **append-only 事件流 + projection 重建** 统一起来了。

关键位置：

- [`backend/src/runtime/session-runtime-service.ts`](../../backend/src/runtime/session-runtime-service.ts)
  - `persistIndex()`
  - `refreshProjection()`
  - `executePlan()`
- [`backend/src/runtime/pi-session-core/session-manager.ts`](../../backend/src/runtime/pi-session-core/session-manager.ts)
  - `_appendEntry()`
  - `appendTodoUpdated()`
  - `appendPauseRequested()`

这意味着后续无论做记忆、compact、history、恢复、任务追踪，都应该优先挂在 runtime 事件流上，而不是回到旧的 snapshot/transcript 口径。

## 4. simple / plan 这条分层仍然是对的

### simple

- 更像单 Agent 对话
- 链路短
- 适合普通问答和轻量任务

### plan

- manager 产 todo
- worker 逐项执行
- runtime 负责 todo、pause、resume 和 projection 刷新

这条边界目前仍然合理，不建议为了做记忆系统而打散。

真正需要补的是：

- plan 的等待/恢复语义更结构化
- worker/manager 的记忆写入入口统一
- todo 与 `foresight` 的边界明确

## 5. 现在最该收敛的 4 个边界

### 5.1 runtime vs session-v2

从现在开始，所有新实施文档和新代码都应默认：

- 主运行时看 `runtime/`
- `session-v2/` 只当兼容/残余参考

### 5.2 session_events vs session_messages

runtime 的真相源是事件流，因此：

- PostgreSQL 一期优先建 `session_events`
- message 只是 event 的子视图

### 5.3 记忆写入入口

当前 [`backend/src/agent/agent-runner.ts`](../../backend/src/agent/agent-runner.ts) 末尾仍走 `recordMemoryTurnAndMaybeFlush()`。

后续要收敛成：

- `MemoryCoordinator.onTurnCompleted()`
- runtime todo 更新额外驱动 `foresight`

### 5.4 shared 协议消费

协议单一事实源已经在 `shared/`，但前端消费还没有完全闭合。后续如果继续增强事件种类，应该优先让前端直接消费 shared 类型，而不是继续在 UI hook 里复制一套 payload 定义。

## 6. 未来两周内与架构最相关的动作

未来两周最重要的不是“大改架构”，而是把主链路钉牢：

1. 对 runtime 做 PostgreSQL dual-write
2. 让 `session_events` 成为记忆系统输入
3. 让 `event memory` 能检索并注入 prompt
4. 再做 compact 原型和 cache-friendly 上下文组织
