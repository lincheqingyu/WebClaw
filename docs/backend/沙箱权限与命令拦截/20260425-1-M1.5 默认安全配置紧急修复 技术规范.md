**触发事件**：Codex M1 落地后，Claude 与用户对真实运行时进行红队检视，发现 `rm -rf /Users/...` 类危险命令在默认配置下**不会触发审批卡**。

---

## 1 问题陈述

`20260424-1` M1 协议层、broker、前端审批卡、hard deny 只读卡均已正确实装。但默认配置下整套机制**不会激活**，根因有二：

### 1.1 `LAYERED_PROMPT` 双职责 flag 默认关闭

权限系统被环境变量 `LAYERED_PROMPT === 'true'` 门控。仓库内：

- `.env` / `.env.example` / `backend/.env` / `release/portable/.env.example` 全部未设置该变量
- 默认 `pnpm dev:full` 启动下，`createPermissionAwareTools` 直接返回原始工具，不经任何包装
- 后端永远不会调用 `ConfirmationBroker.create()`
- 前端 `<ApprovalCard>` 永远不会被触发

后果：M1 验收清单的 7 条核心场景**没有一条**能在默认配置下自然触发。

### 1.2 flag 命名误导

`LAYERED_PROMPT` 字面意思是"分层提示词实验开关"，但其实际控制范围跨两个完全独立的特性：

- **特性 A：权限系统**（本 M1.5 范围内）
  - `agent-runner.ts:92`、`manager-runner.ts:118`、`worker-runner.ts:172`、`permission-manager-registry.ts:48`
- **特性 B：分层 system prompt + memory recall**（不在本范围内）
  - `session-runtime-service.ts:1326 / 1376 / 1455`

把权限作为一个"实验 flag"耦合在另一个特性的开关上，是命名与默认值的双重错误。

### 1.3 不在本规范范围的关联问题

**`bash` 工具裸跑**：`backend/src/agent/tools/bash.ts` 当前是 `execSync(command, { cwd: PROJECT_ROOT })`，未接 `ChildProcessSandbox`，命令字符串中的绝对路径不经 `canEditFile` / `canDeleteFile` 检查。即使 M1.5 完成、权限系统默认生效，`rm -rf /Users/foo/Documents` 仍可能因 bash 分类器覆盖度不足而漏过。

该问题留待 **M1.6**（独立规范）处理，本文不展开。

---

## 2 决策

**删除 `LAYERED_PROMPT` 对权限系统的门控**，权限系统作为产品默认行为永久启用，不保留运行期 flag。

### 2.1 决策表

| 维度 | 决策 | 理由 |
|---|---|---|
| 方案选择 | A：删 flag · 永远开 | 权限不是实验，是产品核心安全特性。保留 flag 只会让"忘记开 = 系统裸奔"成为长期失误源。 |
| 兼容性 | 不保留逃生通道 | Lecquy 不允许用户随便关权限。需要降级请走 `LECQUY_PERMISSION_MODE=acceptAll` 等模式参数（保留），而不是关掉整个系统。 |
| 失败降级 | 创建 `PermissionManager` 失败仍允许工具运行，但通过 logger 输出 ERROR 级日志 | 启动期权限系统挂掉不应让整个 Agent 崩溃，但必须显著告警，避免静默裸跑。 |

### 2.2 不在本次改动范围

- **不动** `session-runtime-service.ts` 的 1326 / 1376 / 1455 三处 `LAYERED_PROMPT` 引用 —— 那是分层 system prompt / memory recall 的独立 flag，与权限系统正交。本次只解耦权限那一面。
- **不动** `LECQUY_PERMISSION_MODE` —— 该环境变量控制 `PermissionManager.create` 的 `initialMode`，与 flag 删除无关，继续保留。
- **不引入新的 flag**（如 `LECQUY_PERMISSIONS_ENABLED`）—— 永久开启等于"无 flag"。
- **不动 bash 工具实现** —— bash 沙箱接入是 M1.6 范围。
