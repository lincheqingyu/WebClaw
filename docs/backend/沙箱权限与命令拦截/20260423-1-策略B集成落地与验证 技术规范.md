## 2 实际实现与设计文档的差异

### 2.1 `manager` 作为可选注入，而非强耦合

`20260420-5` 的样例代码没有区分「未注入 manager」和「注入后失败」两种情况，这里在真实实现中做了**双层兜底**：

```ts
// tool-permission.ts
let bridged: BridgedTier | null = null
if (options.manager) {
  try {
    const result = await options.manager.check({ ... })
    bridged = bridgeResult(result)
  } catch {
    // 新引擎抛错：降级到旧引擎（不 rethrow）
    bridged = null
  }
}

// 未注入 manager 或 manager.check 抛错都落到这里
const legacyTier = classifyToolPermission(...)
const tier = bridged ? mostRestrictiveTier(bridged.tier, legacyTier) : legacyTier
```

**动机**：`checker.ts` 已经在内部用 `try/catch` 把分类器异常翻译成 `deny`，所以 `manager.check` 本身极少抛错。这里额外兜一层是为了防御未来任何在 `bridgeResult` / 其它路径冒出的异常把主流程打挂。

### 2.2 通过环境变量 `LAYERED_PROMPT` 做门闸

没有改已有的 `enabled` 参数语义，而是让 `permission-manager-registry` 自行读取 `LAYERED_PROMPT`：

- `LAYERED_PROMPT !== 'true'` → `getPermissionManager` 直接返回 `undefined` → runner 传入的 `manager` 为 `undefined` → 新引擎完全不介入
- `LAYERED_PROMPT === 'true'` → 首次加载创建单例 → 后续复用

**动机**：与现有「分层 Prompt」特性开关对齐。分层 Prompt 和新权限引擎本来就是同一批能力，统一旗标方便灰度。

### 2.3 `hardDeny` 仍然发 `confirm_required` 事件

`20260420-5` 的样例代码里 `hardDeny` 也发了 `confirm_required`。这里保留了这个行为，原因是：

1. 前端目前只有「tool_state + isError」一种渠道展示被拦截的操作
2. `description` 里有 `"操作已被拒绝"` 前缀，人眼可读地区分「硬拒绝」和「需确认」
3. 短期内不会触发 `hardDeny`（只有路径遍历等极端场景），事件数量可忽略

**隐患**：前端看到事件类型是 `confirm_required`，但实际上这是「无法恢复」的拒绝。前端交互改造时要在 `ConfirmRequiredEvent` 上扩一个 `hardDeny: boolean` 字段（见 `20260423-2` 第 3 节）。

### 2.4 Registry 的 `Promise<null>` 语义

`getPermissionManager` 会把「创建失败的工作区」缓存为 `null`，直到进程重启才重试：

```ts
const managerByWorkspace = new Map<string, Promise<PermissionManager | null>>()
```

这个设计模仿了 Node 对 `require` 失败的处理 —— 避免每次调用都去重试一个注定失败的工作区。生产环境里创建失败大多是配置文件解析错误，重试没意义。

**注意**：热重载或测试需要 `clearPermissionManagerCache(workspaceDir)` 显式清缓存。

## 3 集成测试矩阵

测试文件：`backend/src/agent/__tests__/tool-permission-integration.test.ts`

| # | 场景 | 新引擎 | 旧引擎 | 最终档位 | 验证点 |
| --- | --- | --- | --- | --- | --- |
| 1 | 未注入 manager | - | Auto | Auto | 保持旧引擎行为，`events.length === 0` |
| 2 | deny classifier | hardDeny | Auto | **短路** | 抛 `/操作已被拒绝/`，不走 tool.execute |
| 3 | ask classifier | Confirm | Auto | Confirm | 取更严格，`description` 含分类器原因 |
| 4 | allow classifier + `rm -rf /tmp` | Auto | Confirm | Confirm | 老引擎的 `CONFIRM_PATTERNS` 生效 |
| 5 | `edit_file` + `../../etc/passwd` | hardDeny | Preamble | **短路** | `file-operations` 前置 deny |
| 6 | allow classifier + `ls -la`（关闭 builtin） | Auto | Auto | Auto | 双路放行，不发事件 |
| 7 | `manager.check` 抛错 | - | Confirm | Confirm | 捕获异常降级到旧引擎 |

**关键发现（测试期间补全的坑）**：

- 场景 6 一度失败：`BUILTIN_RULES` 里的 `{bash → ask}` 与分类器的 `allow` 叠加，进入 `resolveByMode` 时被 `matched.behavior === 'ask'` 压倒。**解决办法**：测试里通过 `loadOptions: { includeBuiltin: false, ... }` 旁路 builtin 规则，更贴近"双路都 allow"的语义。这个现象对生产意味着：**线上 bash 的 allow 路径必须配合 rule 放行**，不能只靠分类器。
- 场景 7 的「`manager.check` 自身抛错」无法通过正常 API 触发（`checker.ts` 的 `try/catch` 会把下层异常翻译成 deny）。测试里用解构 + 覆盖的方式构造了一个 `check: async () => throw new Error(...)` 的 broken manager，走 `as unknown as typeof manager` 断言绕过结构检查。

## 4 代码形态速览

### 4.1 双引擎决策骨架

```ts
// backend/src/agent/tool-permission.ts （节选）
export function createPermissionAwareTools(
  tools: readonly AgentTool<any>[],
  options: {
    role: AgentRole
    workspaceDir: string
    enabled: boolean
    onEvent?: (event: ToolPermissionEvent) => void
    manager?: PermissionManager
  },
): AgentTool<any>[] {
  if (!options.enabled) return [...tools]

  return tools.map((tool) => ({
    ...tool,
    execute: async (toolCallId, params, signal, onUpdate) => {
      const args = (params ?? {}) as Record<string, unknown>

      // 1. 新引擎（可选）
      let bridged: BridgedTier | null = null
      if (options.manager) {
        try {
          const result = await options.manager.check({
            toolName: tool.name, args,
            workspaceDir: options.workspaceDir,
            role: options.role,
          })
          bridged = bridgeResult(result)
        } catch {
          bridged = null  // 降级
        }
      }

      // 2. 硬拒绝短路
      if (bridged?.hardDeny) {
        options.onEvent?.({ type: 'confirm_required', ..., description: bridged.description })
        throw new Error(bridged.description)
      }

      // 3. 旧引擎 + 取更严格
      const legacyTier = classifyToolPermission(tool.name, args, options.role, options.workspaceDir)
      const tier = bridged ? mostRestrictiveTier(bridged.tier, legacyTier) : legacyTier

      // 4. 分派（Preamble / Confirm / Auto）
      if (tier === PermissionTier.Preamble) { options.onEvent?.({ ... }) }
      if (tier === PermissionTier.Confirm)  { options.onEvent?.({ ... }); throw new Error(description) }

      return await tool.execute(toolCallId, params, signal, onUpdate)
    },
  }))
}
```

### 4.2 Runner 注入骨架

```ts
// agent-runner.ts / manager-runner.ts / worker-runner.ts 统一写法
const workspaceDir = resolveWorkspaceRoot()
const layeredPermissionEnabled = process.env.LAYERED_PROMPT === 'true'
const permissionManager = await getPermissionManager(workspaceDir)  // 受 LAYERED_PROMPT 门闸

const tools = createPermissionAwareTools(rawTools, {
  role: 'simple' | 'manager' | 'worker',
  workspaceDir,
  enabled: layeredPermissionEnabled,
  manager: permissionManager,
  onEvent: (event) => {
    if (event.type === 'confirm_required') pendingConfirmEvent = event
    onEvent?.(event)
  },
})
```

`pendingConfirmEvent` 在 `getSteeringMessages` 里被消费，强制当前轮次停止继续调用工具、向模型下一条"停"指令。

## 5 对后续工作的约束与暗示

### 5.1 暂未打开的开关

- **`LAYERED_PROMPT=true`**：默认不启用，即生产流量依然走旧引擎。开关打开前建议先在 staging 跑 1 周，对比 `audit-log` 与旧引擎的决策差异。
- **`LECQUY_PERMISSION_MODE`**：未设置时使用配置文件的 `defaultMode`，再退化为 `default`。合法值：`default | dontAsk | plan | acceptEdits | bypassPermissions`。

### 5.2 `hardDeny` 与 `confirm_required` 同渠道

前端当前无法区分「硬拒绝」和「等确认」。详见 [`20260423-2`](./20260423-2-权限确认交互设计%20开发规划.md) 第 3 节的改造方案。

### 5.3 `Confirm` 档位仍依赖模型自我停止

Runner 在收到 `confirm_required` 后通过 `getSteeringMessages` 往对话里塞一条"请停止调用工具"的 user message，让模型下一轮输出收尾文本。这个机制**不保证**模型会乖乖听话 —— 极端情况下模型可能继续试图调用其它工具。保底是 `tracker.iteration` / `tracker.toolFailCount` 的硬上限会把循环拉停。

### 5.4 老规则尚未迁移

`CONFIRM_PATTERNS / PREAMBLE_PATTERNS / AUTO_TOOLS` 还住在 `tool-permission.ts` 里。按 `20260420-5` 第 4 步，它们应当迁移到 `BUILTIN_RULES` 或 `.lecquy/permissions.json`。这是策略 C（新引擎为主）的前置条件，本轮没做。

### 5.5 需要前端介入的事

- 用户审批 / 拒绝 / 本次会话都允许
- 规则编辑器（添加 session / project 规则）
- 审计日志查看
- 权限模式切换（`acceptEdits / plan / dontAsk`）

见下一份文档。
