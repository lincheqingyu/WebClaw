## 问题陈述

策略 B 集成完成后，`confirm_required` 事件通过 WebSocket 抵达前端，目前只被转化成一条 `tool_state` 并标记 `isError: true`。用户**看不到**需要审批的提示，也**无法**批准后继续执行。这让新权限引擎的 `ask` / `Confirm` 档位在实际使用中退化成「拒绝 + 说明」—— 用户只能重新发一次请求、或者开宽规则后重试。

对比参考实现：Claude Code 和 Codex 都在 CLI 侧用**阻塞 stdin** 的方式让用户原地批准并继续。Lecquy 是 Web 架构，需要把这个阻塞语义重写成**异步、可恢复、带会话状态**的 WS 往返。

本文档给出一个可落地的交互协议、前后端分工方案与里程碑。

## 1 Claude Code 与 Codex 是怎么做的

### 1.1 Claude Code（CLI 泄露源码 + 官方文档）

**交互形态**：终端阻塞式。遇到需要确认的工具调用，Claude 在 stdout 打印一个选项菜单，阻塞在 `readline` 上等用户键入。

**典型菜单**：
```
Claude wants to run:
  $ rm -rf node_modules

Approve?
  [y] Yes, run once
  [a] Yes, and allow this tool/pattern for this session
  [A] Yes, and persist to .claude/settings.json (project)
  [n] No, do not run
```

**关键机制**：
1. **决策到达前阻塞**：`permissions.ts` 里 `ask` 决策被返回时，主循环**不往下走**，先调用 `prompt.ts` 拿用户回执
2. **回执转换为规则**：「Allow for session」→ 新增一条 `source: session` 的 allow 规则；「Allow always」→ 调用 `updater.ts` 持久化到 `projectSettings` / `userSettings`
3. **规则匹配前缀**：选择 `a` 时用当前命令的**前缀**作为规则 content，例如批准 `npm install foo` 后 `npm install bar` 自动放行
4. **权限模式切换**：支持 `/mode acceptEdits`、`/mode plan` 等 slash 命令临时切换模式
5. **三档绕过开关**：`--dangerously-skip-permissions`（相当于 `bypassPermissions` 模式）全程不问，但仍 respect deny 规则

**关键实现细节**：
- 决策时机：规则匹配 + preCheck（分类器、文件类）→ `resolveByMode` 产出 `ask/allow/deny` → 若 `ask` 走 prompt
- 持久化：`.claude/settings.json` 用 `deepMerge` 不覆盖用户的其它配置
- 审计：每个决策都写 jsonl；CLI 有 `/audit` slash 命令查看

### 1.2 Codex CLI（OpenAI）

**交互形态**：也是终端阻塞式，但分三档「approval policy」预设：

| 模式 | 文件编辑 | Shell 命令 | 适用场景 |
| --- | --- | --- | --- |
| `suggest`（默认） | 先展示 diff 让人批 | 先展示命令让人批 | 新手、敏感项目 |
| `auto-edit` | 自动执行 | 展示命令让人批 | 熟悉、中等信任 |
| `full-auto` | 自动执行 | 自动执行 | 沙箱环境 |

**菜单**：
```
$ npm install react
[y] Yes   [a] Always in this session   [e] Edit command   [n] No
```

**差异**：
- 没有 Claude Code 那种「持久化到 project / user」层级，会话级粒度更粗
- 有「Edit command」选项，允许用户先改命令再执行 —— 这是 Claude Code 没有的
- `full-auto` 依赖系统沙箱（macOS 的 `sandbox-exec`），拒绝网络 / 拒绝非白名单写路径

### 1.3 两者的共性

1. **每次 ask 都阻塞一次**，用户必须在场
2. **回执不只有 yes/no**，至少有「本次允许 / 本会话允许 / 永久允许」三档
3. **允许时机是"规则"而非"状态"**：批准后会新增规则，下次相同模式命中规则的 allow 分支
4. **拒绝时机是软拒绝**：用户拒绝等同于对话里塞一条"用户拒绝了这个操作"的 assistant-observe message，模型自己决定怎么应对（常见是换种方式继续）

## 2 Lecquy 的 Web 架构带来的新约束

| 维度 | CLI（Claude Code / Codex） | Lecquy（Web + WS） |
| --- | --- | --- |
| 阻塞介质 | 终端 stdin | 不能阻塞 —— 需异步 WS 往返 |
| 用户同在 | 几乎总是 | 用户可能已切页/关网 |
| 并发数 | 1 会话 1 终端 | N 会话并发，每会话多工具并发 |
| 持久化 | 本地文件 | 文件 + 可选上数据库 |
| 设备一致性 | 单机 | 跨设备，需要会话同步 |
| UI 表现力 | 纯文本 | 有 modal / drawer / diff viewer 的余裕 |

**结论**：交互协议必须是**事件驱动 + 带 requestId 关联 + 支持离线重连**的异步往返。参考 Codex 的「approval policy」模式做预设，但不能完全沿用其阻塞 UI。

## 3 建议协议：`confirm_request` / `confirm_response`

### 3.1 服务端事件扩展

现有 `ConfirmRequiredEvent` 扩字段：

```ts
// backend/src/agent/tool-permission.ts
export interface ConfirmRequiredEvent {
  type: 'confirm_required'
  toolCallId: string
  toolName: string
  args: Record<string, unknown>
  description: string
  // 新增 ↓
  requestId: string               // 唯一 id，供前端回执时关联
  kind: 'ask' | 'deny'            // deny 无法批准，仅通知；ask 才有三档回执
  rulePattern?: {
    toolName: string              // 用于"本会话总是允许"建议值
    content?: string              // e.g. "npm install"
  }
  mode: PermissionMode            // 当前权限模式
}
```

### 3.2 新增 `confirm_pending` 会话状态

当工具调用进入 `Confirm` 档时：

1. 不再直接 `throw new Error(description)`
2. 把该轮 agent loop 挂起（见 §4.2 实现方式），把 `confirm_request` 事件透传到前端
3. 会话进入「等待用户审批」状态，前端 UI 渲染审批卡片

### 3.3 客户端回执

前端通过 WS 发回：

```ts
interface ConfirmResponsePayload {
  type: 'confirm_response'
  requestId: string
  decision: 'allow_once' | 'allow_session' | 'allow_project' | 'deny'
  // 仅在 allow_session / allow_project 时生效
  rule?: {
    toolName: string
    content?: string             // 用户可能修改了建议的模式
    description?: string         // 用户备注
  }
}
```

### 3.4 服务端消费回执

- `allow_once`：直接恢复该工具调用的 promise，返回 tool.execute 的结果
- `allow_session`：在 `PermissionManager.applyUpdate` 注入一条 `source: session` 规则，再恢复
- `allow_project`：`applyUpdate` + `persist('projectSettings')`，再恢复
- `deny`：把一条「用户拒绝 {description}」作为 tool result 注入 agent loop，让模型继续处理

### 3.5 `kind === 'deny'` 的退化语义

新引擎 `hardDeny` 时，前端只显示「已拦截」卡片（不提供按钮），后端直接走现有的抛错流程。
这样 UI 能清晰区分「需要你点一下」vs「我已经替你拦了」。

## 4 关键实现决策

### 4.1 会话级「默认审批模式」开关

沿用 Codex 的三档预设，但不叫 approval policy，直接复用现有 `PermissionMode`：

| UI 名称 | 后端 mode | 行为 |
| --- | --- | --- |
| 严谨模式 | `default` | 每次 `ask` 都弹卡片 |
| 仅问 Shell | `acceptEdits` | 编辑 / 读取自动批，shell 仍问 |
| 计划模式 | `plan` | 所有动作只预览不执行 |
| 放开手脚 | `bypassPermissions` | 不问，但仍 respect deny |

**UI 位置**：聊天输入框旁边放一个下拉或分段控件；切换时发 WS `set_permission_mode` 消息；服务端调用 `PermissionManager.setMode`。

### 4.2 挂起 `agent loop` 的两种实现

**方案 A：Promise 挂起（推荐）**

```ts
// tool-permission.ts
const pendingConfirmations = new Map<string, (r: ConfirmDecision) => void>()

if (tier === PermissionTier.Confirm) {
  const requestId = randomUUID()
  options.onEvent?.({ type: 'confirm_required', requestId, kind, ... })

  const decision = await new Promise<ConfirmDecision>((resolve, reject) => {
    pendingConfirmations.set(requestId, resolve)
    signal?.addEventListener('abort', () => {
      pendingConfirmations.delete(requestId)
      reject(new Error('审批被中断'))
    })
  })

  if (decision.verdict === 'deny') throw new Error('用户拒绝')
  if (decision.verdict === 'allow_session') options.manager?.applyUpdate(...)
  // 继续执行
}

// 外部通过 sessionRuntime.resolveConfirm(requestId, decision)
```

- 优点：对 agent loop 透明，不需要改 runner 的状态机
- 缺点：挂起期间 tool 的 `execute` promise 一直在栈里 —— 长时间挂起会占内存；需要配 TTL 超时 + abort signal 清理

**方案 B：显式暂停 / 恢复**

在 `AgentRuntimeEvent` 里加 `agent_paused` / `agent_resumed` 事件，runner 主动 return 当前结果。用户批准后，服务端重建 runner 继续跑。

- 优点：无长挂 promise，重启/迁移友好
- 缺点：改动大，所有 runner 要新增「断点续跑」能力

**建议**：一期用方案 A，TTL = 10 分钟；二期若发现长时间挂起太多，再迁移到方案 B。

### 4.3 规则建议的生成

批准「本会话总是允许」时，前端需要一个合理的规则建议作为默认值：

```ts
function suggestRule(toolName: string, args: Record<string, unknown>): RuleSuggestion {
  if (toolName === 'bash') {
    const command = String(args.command ?? '')
    // 取第一个词作为前缀建议，e.g. "npm install foo" → "npm install"
    const parts = command.split(/\s+/).slice(0, 2).join(' ')
    return { toolName: 'bash', content: parts }
  }
  if (toolName === 'edit_file' || toolName === 'write_file') {
    const filePath = args.path ?? args.file_path
    // 取父目录做 glob 建议
    return { toolName, content: `${path.dirname(String(filePath))}/**` }
  }
  return { toolName }
}
```

前端要允许用户**编辑这个建议**再提交 —— 类似 Claude Code 里的「pattern you want to always allow」输入框。

### 4.4 断线重连

- 前端重连时，WS 先发 `resubscribe_session`
- 服务端查 `pendingConfirmations` 里属于该会话的项，重发所有 `confirm_required` 事件
- 前端重新渲染审批卡片
- 保留 TTL，超时后自动走 `deny` 路径

### 4.5 多并发工具调用

某些场景下（manager → worker）会有多个工具并发。协议上每个 `confirm_required` 有独立 `requestId`，前端 UI 允许**批量审批**：

- 一张卡片列出该轮所有 pending 项
- 「全部批准」按钮复用同一个 rulePattern（如果相同）

## 6 里程碑

### M1（~2d）：协议落地 + 简易审批

- 后端扩 `ConfirmRequiredEvent` + `pendingConfirmations` + resolve API
- 前端最小审批卡片（只有「批准一次」和「拒绝」两个按钮）
- 不做规则持久化，不做模式切换

**验收**：故意触发 `rm -rf /tmp`，前端看到卡片；点批准 → 命令真的执行；点拒绝 → 模型在下一轮说明无法继续。

### M2（~3d）：会话级规则 + 权限模式

- 「本会话总是允许」分支，复用 `PermissionManager.applyUpdate`
- 权限模式切换 UI + 后端 `setMode`
- 断线重连重发 pending

**验收**：批准一次 `npm install foo` 后，再来 `npm install bar` 自动通过；切到 `acceptEdits` 后编辑类工具不再弹。

### M3（~4d）：持久化 + 管理 UI

- 「持久化到 project / user」分支
- 规则编辑器 MVP
- 审计日志查看器 MVP

**验收**：批准并持久化的规则在 `.lecquy/permissions.json` 里落盘；从规则编辑器能看到并删除。

### M4（~2d）：打磨

- 批量审批
- 拒绝时的 agent 收尾文本质量
- 性能 / 内存（长挂 promise 是否泄漏）
- 端到端测试补全

**总计**：约 11 工作日（后端 6.5 + 前端 8 = 14.5 人日，可双线并行 → 自然日约 11）。

## 7 风险与开放问题

### 7.1 长挂 Promise 的资源占用

一个会话挂 1 个 pending 是正常的；如果 manager + N 个 worker 同时各挂 1 个，再乘以同时在线的会话数，可能撑到几百 promise 长期挂在那里。

**缓解**：
- `pendingConfirmations` 用 TTL + 弱引用；超时默认 10 分钟
- 观测：Prometheus 埋 `confirmation_pending_count` 指标
- 极端情况：超过水位线时拒绝新的 `Confirm` 档，直接 deny

### 7.2 同一用户多设备

用户在 A 设备发起请求，B 设备批准。是否允许？

**建议**：M3 之后引入「会话粒度的 pending 列表」，任一订阅该会话的 WS 客户端都能批准。需要考虑「会话所有权」的鉴权（参考现有 `peerId`）。

### 7.3 恶意前端伪造 `confirm_response`

WS 消息要经过现有鉴权链，`requestId` 只在服务端生成，前端拿不到别的会话的 id。但仍需在 resolve 时校验「该 requestId 属于该 peer 当前会话」。

### 7.4 `plan` 模式下的 UX

`plan` 会让所有动作都拦下来只预览。如果一次规划里有 20 个工具调用，弹 20 张卡片体验很差。

**建议**：`plan` 模式下**不发 `confirm_required`**，改发 `plan_preview` 事件，前端汇总成一张"计划草稿"卡片，用户看完后点"开始执行"才离开 plan 模式。

### 7.5 模型会不会在批准后仍然走老路

`allow_once` 恢复后，模型下一轮不会再次触发相同 check（同一次 tool call 已经拿到结果）。但如果之后模型再次发出相同命令，会重新走一次决策 —— 这是正确行为。UI 上要让用户明白「本次允许」和「会话允许」的差异。

## 9 决策点：要不要做？

**强烈建议做**，理由：

1. 当前 `confirm_required` 只会把用户请求「软拒绝」掉，严格意义上新权限引擎处于不可用状态。用户会把 `LAYERED_PROMPT` 关掉回退老引擎，Opus 交付的 2757 行就白费了。
2. 前端交互是使用新引擎的唯一出口；没有它，规则编辑、模式切换、审计查看全都只能改文件。
3. M1 的工作量只有 ~2d，收益是从「不可用」到「可用」。

**如果暂不做**，建议至少：

- 在 UI 上把 `confirm_required` 的 `tool_state` 文案改成「此操作被拦截，需要放宽规则后重试。编辑 `.lecquy/permissions.json` 即可」
- 保持 `LAYERED_PROMPT` 默认关闭，仅在内部测试开启
