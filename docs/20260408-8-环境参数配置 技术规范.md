# Lecquy 环境参数配置说明

本文档说明 Lecquy 当前环境变量的含义、可选项、生效范围、修改后果，以及推荐的配置方式。

## 1. 配置入口与生效范围

### 根目录 `.env` 是本地开发主入口

- 后端启动时会从项目根目录加载 `.env`
- 前端 Vite 开发/构建时也会从项目根目录读取 `.env`
- 根目录 `.env.example` 是推荐模板

也就是说，本地开发优先维护：

```txt
Lecquy/.env
Lecquy/.env.example
```

### `backend/.env` 不是当前本地运行主入口

当前源码里的后端启动入口读取的是根目录 `.env`，不是 `backend/.env`。  
但 `deploy/` 目录下的一些历史部署脚本仍沿用 `backend/.env` 约定，所以如果你在做部署脚本改造，需要额外留意这两套入口不要混淆。

## 2. 参数优先级

当前配置大致按下面顺序生效：

1. 前端设置 / WebSocket 请求体传入的运行参数
2. `.env` 中的环境变量
3. 代码里的默认值
4. 兼容旧版本的 `VITE_*` 变量兜底

### 会覆盖 `.env` 的运行时参数

以下参数支持由前端设置或请求体覆盖，不一定总是使用 `.env`：

- `model` 覆盖 `LLM_MODEL`
- `baseUrl` 覆盖 `LLM_BASE_URL`
- `apiKey` 覆盖 `LLM_API_KEY`
- `options.maxTokens` 覆盖 `LLM_MAX_TOKENS`
- `options.temperature` 直接作为本次请求温度使用

这意味着：`.env` 更像“默认值”，而不是绝对值。

## 3. 推荐最小配置

日常本地开发通常只需要这几项：

```bash
HOST=0.0.0.0
BACKEND_PORT=3011
FRONTEND_PORT=5173
LLM_API_KEY=your-api-key
LLM_MODEL=glm-4-plus
```

只有在前后端不走同源地址时，才需要额外加：

```bash
BACKEND_ORIGIN=http://your-backend-host:3011
```

## 4. 修改后何时生效

- 修改后端相关环境变量后，需要重启后端进程
- 修改前端地址相关变量后，需要重启 Vite 开发服务
- 生产环境下，前端构建产物依赖构建时环境变量，修改后需要重新构建前端

## 5. 参数清单

### 5.1 共享网络参数

| 参数 | 默认值 / 是否必填 | 可选项 / 格式 | 修改后果 | 建议 |
|------|-------------------|---------------|----------|------|
| `HOST` | 默认 `0.0.0.0` | 任意非空字符串，常见值：`0.0.0.0`、`127.0.0.1`、局域网 IP | 同时影响后端监听地址和前端 Vite dev server 监听地址。改成 `127.0.0.1` 后，本机外设备无法访问；改成错误 IP 可能直接导致服务无法监听。 | 本地联调用 `0.0.0.0`，只想本机访问时用 `127.0.0.1`。 |
| `BACKEND_PORT` | 默认 `3000` | `1-65535` 的整数 | 后端实际监听端口会变化，前端默认推导的 API/WS 地址也会跟着变。若被占用，服务启动失败。 | 本地建议显式写入，避免多人协作时口径不一致。 |
| `FRONTEND_PORT` | 默认 `5173` | `1-65535` 的整数 | 影响前端 Vite dev / preview 的展示端口。若被占用，前端开发服务启动失败；改动后需要用新端口重新访问页面。 | 本地建议显式写入，和后端端口分开维护。 |
| `BACKEND_ORIGIN` | 可选，无默认值 | 绝对 `http://` 或 `https://` 地址，例如 `http://192.168.1.10:3011` | 前端会优先用它作为 API 基地址，并自动推导 WS 地址。填错会导致前端请求到错误的后端；若误填成 `ws://`，HTTP API 会失效。 | 只有当前后端不共用同一个页面来源时才需要配置。 |
| `VITE_DEV_HOST` | 兼容旧配置，不推荐新写 | 与 `HOST` 类似 | 仅作为旧变量兜底，效果被 `HOST` 覆盖。继续使用会增加维护成本。 | 新配置统一改用 `HOST`。 |
| `VITE_API_BASE` | 兼容旧配置，不推荐新写 | 绝对 HTTP(S) 地址 | 仅作为旧变量兜底。若同时配置了 `BACKEND_ORIGIN`，优先使用 `BACKEND_ORIGIN`。 | 新配置统一改用 `BACKEND_ORIGIN`。 |
| `VITE_WS_BASE` | 兼容旧配置，不推荐新写 | 绝对 WS(S) 地址 | 只影响前端 WS 地址。若与 HTTP 基地址不一致，调试时容易出现“HTTP 正常、WS 失败”的分裂问题。 | 仅保留兼容，不建议继续新增。 |

### 网络参数特别说明

- `BACKEND_ORIGIN` 是“单一覆盖入口”，目的就是避免再分别维护 HTTP 和 WS 两个地址
- 未配置 `BACKEND_ORIGIN` 时，前端会按“当前页面域名 + `BACKEND_PORT`”自动推导目标后端
- 这套自动推导很适合同源部署和局域网调试，但不适合前后端分开部署的场景

### 5.2 运行时与 LLM 参数

| 参数 | 默认值 / 是否必填 | 可选项 / 格式 | 修改后果 | 建议 |
|------|-------------------|---------------|----------|------|
| `NODE_ENV` | 默认 `development` | `development` / `production` / `test` | `production` 时，如果 `frontend/dist` 存在，后端会托管前端静态文件并承担 SPA 回退；设错可能导致静态资源托管行为与预期不一致。 | 本地开发用 `development`，生产部署用 `production`。 |
| `LOG_LEVEL` | 默认 `info` | `debug` / `info` / `warn` / `error` | 控制日志量。`debug` 会显著增加输出，排障方便但会淹没终端；`error` 太高会丢失很多上下文。 | 日常开发用 `info`，问题排查临时切到 `debug`。 |
| `LLM_API_KEY` | 必填 | 任意非空字符串 | 未配置时后端启动校验直接失败；配置错误时模型请求会返回鉴权错误。 | 必填，建议只放服务端默认 key。 |
| `LLM_BASE_URL` | 默认 `https://open.bigmodel.cn/api/paas/v4/` | OpenAI 兼容接口的绝对 URL | 决定模型请求发往哪里。改错会导致 404、鉴权失败或协议不兼容。 | 仅在切换 OpenAI 兼容供应商、代理层或自建 vLLM 时修改。 |
| `LLM_MODEL` | 后端默认 `glm-4.7` | 任意模型 ID 字符串 | 会影响默认模型选择。模型名错误会导致接口报错；换成长上下文/视觉模型可能改变能力、成本和响应速度。 | 项目当前示例推荐 `glm-4-plus`，如果你不想依赖代码默认值，建议显式写入。 |
| `LLM_MAX_TOKENS` | 默认 `8192` | 正整数 | 决定单次输出的最大 token 上限。过小会截断回答；过大可能增加成本、时延，且部分模型会拒绝超限值。 | 一般按模型能力设置，日常 4k-8k 足够。 |
| `LLM_TEMPERATURE` | Schema 默认 `0.7`，但当前默认调用链未读取 | `0-2` 浮点数 | 当前版本里，绝大多数请求使用的是前端/请求体传入的温度，而不是这个环境变量；单改它通常不会产生预期效果。 | 暂时把它视为“保留参数”，不要把全局控温依赖在它上面。 |
| `LLM_TIMEOUT` | Schema 默认 `120000`，但当前默认调用链未读取 | 毫秒整数 | 当前版本里没有把它接入主模型调用超时控制，单改它通常不会影响请求时长。 | 暂时把它视为“保留参数”，如要真正生效需要补代码。 |
| `VITE_USE_PI_WEB_UI_PARTIAL` | 默认 `false` | `true` / `false` 字符串 | 为 `true` 时，前端消息列表和输入框会切换到 `PiMessageListAdapter` / `PiChatInputAdapter`。若适配层未覆盖完整功能，可能出现交互差异。 | 仅在验证 Pi Web UI 局部适配时开启。 |

### LLM 参数特别说明

- 当前“代码默认值”和“示例推荐值”并不完全一致：
  - 后端环境默认模型是 `glm-4.7`
  - `.env.example` 当前推荐显式写 `glm-4-plus`
- 如果你不想被代码默认值影响，最稳妥的方式是显式配置 `LLM_MODEL`
- `LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL` 都可能被前端请求覆盖，所以生产环境若不希望用户自带 key/模型，需要另做服务端限制

### 5.3 会话与上下文裁剪参数

这些参数都已经有默认值，只有在你明确知道自己要调整会话生命周期或上下文体积时才建议改。

| 参数 | 默认值 | 可选项 / 格式 | 修改后果 | 建议 |
|------|--------|---------------|----------|------|
| `SESSION_MAIN_KEY` | `main` | 任意字符串 | 影响特定 DM 会话是否被识别成“主会话”键名。改动后，旧会话 key 规则会变化，可能造成看起来像“历史会话丢了，实际是 key 变了”。 | 除非你明确在做多入口会话路由设计，否则不要改。 |
| `SESSION_RESET_MODE` | `daily` | `daily` / `idle` | `daily` 模式下会按每日边界重置，并且空闲超时也会触发重置；`idle` 模式只按空闲时间重置。切换后会直接改变“什么时候开启新会话”的规则。 | 大多数场景保持 `daily` 即可。 |
| `SESSION_RESET_AT_HOUR` | `4` | `0-23` 整数 | 仅在 `daily` 模式下有意义，表示本地时区每天几点作为“新会话日界线”。改早了会更频繁切会话，改晚了会更长时间沿用旧上下文。 | 适合按团队工作日节奏微调，例如凌晨 4 点或早上 6 点。 |
| `SESSION_IDLE_MINUTES` | `120` | 大于等于 `1` 的整数 | 超过这个空闲时间后，即使仍在 `daily` 模式，也会切新会话。设太小会话会被切得很碎；设太大旧上下文会残留更久。 | 1-3 小时通常比较稳。 |
| `SESSION_STORE_DIR` | `.lecquy/sessions/v3` | 工作区内相对路径 | 决定会话快照和索引存储位置。改动后新旧数据会分散到不同目录，看起来像“切换到新库”。路径越界会被拒绝。 | 除非要迁移存储目录，否则保持默认。 |
| `SESSION_PRUNING_MODE` | `cache-ttl` | `off` / `cache-ttl` | `off` 完全不裁剪，长会话会不断膨胀；`cache-ttl` 会在一定条件下裁剪旧工具输出，减少上下文体积。 | 正常使用建议保留 `cache-ttl`。 |
| `SESSION_PRUNING_TTL` | `5m` | `500ms`、`30s`、`10m`、`2h` 这类格式 | 表示距离上次模型调用超过多久后，才允许重新检查裁剪。设太短会更频繁裁剪；设太长则旧大结果保留更久。格式非法时当前实现会静默回退到 `5m`。 | 一般保持 `5m` 或 `10m`。 |
| `SESSION_PRUNING_KEEP_LAST_ASSISTANTS` | `3` | 大于等于 `0` 的整数 | 表示最近多少条 assistant 回复之前的旧 `toolResult` 才允许被裁剪。设太小可能把仍有参考价值的工具结果提前裁掉；设太大则裁剪不明显。 | 2-4 通常比较合适。 |
| `SESSION_PRUNING_SOFT_RATIO` | `0.3` | `0-1` 浮点数 | 达到软阈值时，旧工具结果会被“截头去尾保留摘要”。设太低会过早摘要；设太高则保留大块原始输出。 | 先保持默认，除非上下文成本明显过高。 |
| `SESSION_PRUNING_HARD_RATIO` | `0.5` | `0-1` 浮点数 | 达到硬阈值时，旧工具结果会被完全清空成占位文本。设太低会明显损失可追溯性；设太高则大文本仍会压上下文。 | 必须大于等于软阈值附近才合理，建议不要低于 `SESSION_PRUNING_SOFT_RATIO`。 |
| `SESSION_PRUNING_MIN_TOOL_CHARS` | `50000` | 大于等于 `1` 的整数 | 只有超过这个字符数的旧工具结果才会参与裁剪。设太低会频繁动小结果；设太高又起不到减负作用。 | 只有工具结果经常非常大时才需要下调。 |

### 会话裁剪特别说明

- 裁剪只针对“较旧的 `toolResult`”，不会无差别裁普通用户消息
- 带图片块的工具结果不会被裁掉
- `SESSION_PRUNING_SOFT_RATIO` 和 `SESSION_PRUNING_HARD_RATIO` 是相对于内部上下文窗口估算值计算的，不是绝对 token 数
- `SESSION_RESET_MODE=daily` 并不代表“只看日期”，它仍然会叠加 `SESSION_IDLE_MINUTES` 的空闲重置逻辑

### 5.4 PostgreSQL 参数

这些参数用于记忆系统的 PostgreSQL 底座。**默认关闭**，只有显式开启后才会在服务启动时连接数据库并执行 migration。

| 参数 | 默认值 / 是否必填 | 可选项 / 格式 | 修改后果 | 建议 |
|------|-------------------|---------------|----------|------|
| `PG_ENABLED` | 默认 `false` | `true` / `false` | 关闭时服务完全不接触 PostgreSQL；开启后启动阶段会连接数据库并执行 migration，连不上会直接启动失败。 | 本地未装 PostgreSQL 时保持 `false`；开始 dual-write 或 memory 开发时再打开。 |
| `PG_HOST` | 默认 `localhost` | 主机名或 IP | 改错会导致连接超时。 | 本地开发保持默认即可。 |
| `PG_PORT` | 默认 `5432` | `1-65535` 整数 | 改错会导致连接失败。 | PostgreSQL 默认端口 5432，通常无需修改。 |
| `PG_DATABASE` | 默认 `lecquy` | 任意字符串 | 改变后会连接到不同的数据库，历史数据不可见。 | 建议保持 `lecquy`，数据库需提前创建。 |
| `PG_USER` | 默认 `postgres` | 任意字符串 | 影响连接权限，用户需有建表权限。 | 本地开发可用默认 postgres 用户。 |
| `PG_PASSWORD` | 可选，无默认值 | 任意字符串 | 未填且数据库要求密码时会鉴权失败。 | 本地若用 trust 认证可不填；生产环境必填。 |
| `PG_POOL_MAX` | 默认 `10` | 大于等于 `1` 的整数 | 过小会在并发高时排队等待；过大会超出 PG 连接上限。 | 本地开发保持默认即可；生产环境视并发量调整。 |
| `PG_SSL` | 默认 `false` | `false` / `true` / `require` | `false` 不启用 SSL；`true` 启用并验证证书；`require` 启用但跳过证书验证（适合云托管服务）。 | 本地开发用 `false`；云数据库（如 Supabase、RDS）通常用 `require`。 |

### PostgreSQL 参数特别说明

- 一期 PG 已覆盖连接池、migration、runtime dual-write、memory 写入与 RAG 实验表
- 只有 `PG_ENABLED=true` 时，`server.ts` 才会在启动时执行 migration，并在优雅关闭时释放连接池
- `pnpm db:migrate` 会自动创建当前已落地的 runtime / memory / knowledge 表
- migration 幂等：已执行的版本会被跳过，可安全重复运行
- 数据库须提前手动创建（`CREATE DATABASE lecquy`）；migration 不会自动创建数据库

### 本地 PG 验收环境（已在当前机器真实验证）

当前仓库已经补了一个可重复启动的本地 PostgreSQL 验收入口：

```bash
pnpm pg:dev:start
pnpm pg:dev:stop
pnpm pg:dev:status
```

当前已验证方案依赖：

- Homebrew `postgresql@16`
- 仓库内数据目录 `/.lecquy/dev-postgres/`

当前默认连接参数：

```bash
PG_ENABLED=true
PG_HOST=127.0.0.1
PG_PORT=5432
PG_DATABASE=lecquy
PG_USER=postgres
PG_PASSWORD=
PG_SSL=false
```

推荐验收顺序：

```bash
pnpm pg:dev:start
PG_ENABLED=true PG_HOST=127.0.0.1 PG_PORT=5432 PG_DATABASE=lecquy PG_USER=postgres PG_PASSWORD='' PG_SSL=false LLM_API_KEY=pg-smoke-key LLM_MODEL=pg-smoke-model pnpm --filter @lecquy/backend run db:migrate
PG_ENABLED=true PG_HOST=127.0.0.1 PG_PORT=5432 PG_DATABASE=lecquy PG_USER=postgres PG_PASSWORD='' PG_SSL=false LLM_API_KEY=pg-smoke-key LLM_MODEL=pg-smoke-model pnpm --filter @lecquy/backend run pg:smoke
pnpm pg:dev:stop
```

`pg:smoke` 当前会真实验证：

- migration
- `sessions / session_events`
- `memory_items(kind='event')`
- `memory_items(kind='foresight')`
- compaction event
- `knowledge_documents / knowledge_chunks` ingest 与 search

当前不会覆盖：

- 前端 / WS 端到端
- 真实上游 LLM 提取质量

说明：

- event extraction smoke 为了避免依赖外部 LLM，会强制走“快速失败 -> heuristic fallback”
- 但最终 `memory_items(kind='event')` 仍然是真实 PostgreSQL 落库
- 当前机器没有可用 `docker`，所以本轮没有补一份经过实机验证的 Compose 启动结果

### 前端同协议 WS 验收入口

当前仓库还补了一个 dev-only 的 WS 验收脚本：

```bash
pnpm --filter @lecquy/backend run pg:ws-acceptance
```

推荐与独立验收 backend 实例一起使用：

```bash
HOST=127.0.0.1 BACKEND_PORT=3012 BACKEND_ORIGIN=http://127.0.0.1:3012 \
LLM_BASE_URL=http://192.168.3.49:8003/v1 \
LLM_MODEL=qwen3-coder-30b-instruct \
LLM_API_KEY=local-vllm-key \
PG_ENABLED=true PG_HOST=127.0.0.1 PG_PORT=5432 PG_DATABASE=lecquy PG_USER=postgres PG_PASSWORD='' PG_SSL=false \
pnpm --filter @lecquy/backend exec tsx src/server.ts
```

另开一个终端执行：

```bash
HOST=127.0.0.1 BACKEND_PORT=3012 BACKEND_ORIGIN=http://127.0.0.1:3012 \
LLM_BASE_URL=http://192.168.3.49:8003/v1 \
LLM_MODEL=qwen3-coder-30b-instruct \
LLM_API_KEY=local-vllm-key \
PG_ENABLED=true PG_HOST=127.0.0.1 PG_PORT=5432 PG_DATABASE=lecquy PG_USER=postgres PG_PASSWORD='' PG_SSL=false \
pnpm --filter @lecquy/backend run pg:ws-acceptance
```

当前真实结果：

- `simple + WebSocket + sessions / session_events + event memory + compaction` 已验证通过
- `plan -> foresight` 当前仍被模型侧阻塞：本地 `qwen3-coder-30b-instruct` 在真实 WS 验收里连续两次生成 `todo_updated.items = []`，因此不会写入 `memory_items(kind='foresight')`
- 这条结果代表“前端同协议客户端已验收”，不代表浏览器 DOM / UI 自动化已验收

### 可达 LLM event extraction 验收入口

当前仓库还补了一个 live extraction 验收脚本：

```bash
pnpm --filter @lecquy/backend run pg:live-extraction
```

使用可达本地模型端点时可直接运行：

```bash
LLM_BASE_URL=http://192.168.3.49:8003/v1 \
LLM_MODEL=qwen3-coder-30b-instruct \
LLM_API_KEY=local-vllm-key \
PG_ENABLED=true PG_HOST=127.0.0.1 PG_PORT=5432 PG_DATABASE=lecquy PG_USER=postgres PG_PASSWORD='' PG_SSL=false \
pnpm --filter @lecquy/backend run pg:live-extraction
```

## 6. 常见配置方案

### 方案 A：最小本地开发

```bash
HOST=0.0.0.0
BACKEND_PORT=3011
FRONTEND_PORT=5173
LLM_API_KEY=your-api-key
LLM_MODEL=glm-4-plus
```

适用场景：

- 前后端都在本机开发
- 页面和后端走同一台机器
- 不需要数据库或档案扩展

### 方案 B：局域网联调

```bash
HOST=0.0.0.0
BACKEND_PORT=3011
FRONTEND_PORT=5173
LLM_API_KEY=your-api-key
LLM_MODEL=glm-4-plus
```

特点：

- 手机、平板、同网段电脑可以访问
- 前端会按访问时使用的主机名自动推导后端地址

### 方案 C：前后端分离部署

```bash
HOST=0.0.0.0
BACKEND_PORT=3011
FRONTEND_PORT=5173
BACKEND_ORIGIN=https://api.example.com
LLM_API_KEY=your-api-key
LLM_MODEL=glm-4-plus
```

特点：

- 页面可以部署在别的域名
- 前端不会再按当前页面地址猜后端，而是固定请求 `BACKEND_ORIGIN`

## 7. 不建议直接依赖的参数

以下变量当前更适合作为“兼容或预留”，不建议把核心配置依赖建在它们上面：

- `VITE_DEV_HOST`
- `VITE_API_BASE`
- `VITE_WS_BASE`
- `LLM_TEMPERATURE`
- `LLM_TIMEOUT`

原因：

- 前 3 个已被新参数体系替代
- 后 2 个虽然在 schema 中声明了，但当前默认调用链并没有真正把它们当作全局默认值使用

## 8. 排查建议

如果改完参数后出现问题，优先按下面顺序排查：

1. 先确认改的是根目录 `.env`，不是误改 `backend/.env`
2. 改完后确认对应服务是否已经重启
3. 前端连不上后端时，先看 `BACKEND_ORIGIN` 是否误填成 `ws://`
4. 请求发出但模型报错时，优先检查 `LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL`
5. 会话“突然新开”时，重点检查 `SESSION_RESET_MODE`、`SESSION_RESET_AT_HOUR`、`SESSION_IDLE_MINUTES`
6. 长会话内容“被清掉”时，重点检查 `SESSION_PRUNING_*`
