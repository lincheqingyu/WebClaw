# Lecquy 最终方向草案：个人强 Agent

更新日期：2026-04-01

## 1. 这次重新定义的核心结论

Lecquy 的最终方向，不再只是“AI Web 对话客户端”，而是演进成一个 **面向个人、具备持续记忆、可主动执行任务、能连接真实网络世界的强 Agent**。

当前阶段可以认为，项目已经大致具备了 2025 年常见 AI Web 产品的一些基础能力；后续工作的重点，不是继续堆普通聊天体验，而是把系统往“长期陪伴、长期记忆、长期执行”的方向拉升。

一句话版本：

> Lecquy 要做的，是一个以个人为中心的长期 Agent 系统，而不是一个只会聊天的网页壳。

## 2. 产品定位

### 2.1 目标形态

最终希望形成的产品形态：

- 有长期记忆，能够记住用户、任务、偏好、上下文和过去行动
- 能访问外部网络，补足模型静态知识的不足
- 能像一个真正的代理一样，持续跟进、自动执行、主动提醒
- 不只回答问题，还能管理信息、组织任务、沉淀资产
- 尽量服务“个人强使用者”场景，而不是先追求通用社交化产品

### 2.2 与当前项目的关系

当前已有的对话、会话、工具调用、前后端联动能力，不是终点，而是底座。后面的方向应该围绕下面几个关键词统一推进：

- Personal-first
- Agent-native
- Memory-centric
- Action-oriented
- Local-first / privacy-aware

## 3. 后续强化方向

### 3.1 方向一：AI Agent 记忆

这是当前最重要的强化方向，优先级最高。

原因：

- 个人强 Agent 如果没有长期记忆，就只能停留在“单轮聪明”
- 自动任务、长期协作、个性化服务，都建立在记忆系统之上
- 记忆会直接决定产品是否具备“越来越懂你”的核心壁垒

### 3.1.1 想重点关注的能力

- 长期记忆：跨会话保留人物、事实、偏好、项目状态
- 任务记忆：记住待办、计划、进行中的工作、历史决策
- 情境记忆：记住“为什么当时这么做”，而不是只记结论
- 多模态记忆：未来支持截图、界面、文档片段等视觉记忆
- 记忆治理：写入、检索、压缩、遗忘、冲突处理、重要性排序
- 本地优先：尽量把关键记忆掌握在本地和项目自身体系里

### 3.1.2 你当前记录的参考项目

说明：下面的指标与备注是你这次思考里记录下来的候选信息，后续正式调研时再逐项核验。

| 项目 | 你记录的信息 | 当前可借鉴点 |
|------|---------------|--------------|
| EverMemOS | LoCoMo 93.05%，LongMemEval 83.00%，MIT，当前最高 | 高质量长期记忆系统设计 |
| MIRIX | LoCoMo 85.38%，论文开源，多模态，截图记忆 | 多模态记忆与视觉上下文 |
| Hindsight | 89.61% / 91.4%，MIT，最高 | 高质量回溯与事件记忆 |
| SuperLocalMemory (Mode C) | LongMemEval 87.7%，纯本地，零云依赖 | 本地优先的记忆方案 |
| Letta | ~83%，Apache 2.0，全 agent runtime | 记忆与 agent runtime 的一体化 |
| Mem0 | 65.99%，Apache 2.0，最大社区 | 工程落地成熟度与生态参考 |
| Zep / Graphiti | 75.14%，Apache 2.0，时序图谱强 | 时间序列图谱与关系记忆 |

### 3.1.3 这条线的落地思路

建议不要一开始就追求“最强论文复刻”，而是分三层逐步演进：

1. 先把 Lecquy 自己的记忆模型定义清楚
   - 用户画像记忆
   - 项目/任务记忆
   - 会话摘要记忆
   - 可检索原始事件流
2. 再补记忆机制
   - 何时写入
   - 如何抽取
   - 如何检索
   - 如何压缩
   - 如何纠错与更新
3. 最后再追求高级形态
   - 多模态截图记忆
   - 图谱化关系记忆
   - 本地长期记忆优化
   - 自动反思与记忆重整

### 3.2 方向二：Web Search

这是第二优先级方向。

原因：

- 记忆解决“过去我知道什么”，搜索解决“现在外部世界发生了什么”
- 没有搜索，Agent 很难具备现实世界可用性
- 搜索不仅是补充答案，更是任务执行、信息核验、网页抓取、RAG 的基础能力

### 3.2.1 目标能力

- 全网检索
- 搜索结果去噪、重排、摘要
- 特定网页抓取与结构化抽取
- 搜索结果和记忆系统联动
- 搜索结果沉淀为可复用知识

### 3.2.2 你当前记录的参考方案

| 方案 | 你记录的特点 | 适合场景 |
|------|--------------|----------|
| SearXNG | 零额外成本（除服务器），难度较低 | 任何 agent 的搜索后端，高频查询 |
| Perplexica | 零额外成本，接近 Perplexity 体验 | 想做带 AI 综合回答的搜索 |
| AgenticSeek | 零额外成本（含本地 LLM） | 完全离线、隐私优先的自主 agent |
| Crawl4AI + SearXNG | 成本极低，但需 embedding API | 需要 RAG + 网搜一体的复杂场景 |
| OpenCLI | 零额外成本 | 特定网站的结构化数据抓取，不是全网搜索 |

### 3.2.3 当前建议

这一方向更适合走“分层能力”：

- 第一层：先做稳定搜索后端，例如 SearXNG
- 第二层：再做搜索结果摘要与引用
- 第三层：补网页抓取、结构化抽取、RAG 联动
- 第四层：再考虑接近 Perplexity 式的一体化搜索体验

也就是说，**不要一开始把 Web Search 理解为一个 UI 功能，而要理解为 Agent 的基础设施能力。**

### 3.3 方向三：模仿 OpenClaw 的 Agent 自动任务

这是第三优先级方向。

目标不是“定时跑脚本”，而是让 Agent 从被动响应升级到主动工作。

### 3.3.1 理想能力

- 自动接收任务并拆解
- 定时或触发式执行
- 对长期任务持续跟进
- 自动生成阶段性结果、总结、提醒
- 在需要时向用户确认，而不是所有事情都全自动
- 让用户形成“把事情交给它，它会记着并继续推进”的信任感

### 3.3.2 与前两条线的关系

自动任务本身不是孤立能力，它会强依赖前两项基础设施：

- 没有记忆，自动任务无法长期持续
- 没有搜索，自动任务无法接触外部世界

所以这条线更像是前两项成熟之后的“价值放大器”。

### 3.3.3 这条线的核心价值

如果这部分做成，Lecquy 才会从“会回答”变成“会持续做事”。

这也是最接近“个人强 Agent”最终体验的一步。

### 3.4 方向四：暂时保留开放位

第四条线目前先不强行定义，这是合理的。

原因：

- 当前最关键的产品势能已经来自前三条主线
- 现在就给第四条线下结论，容易把路线拉散
- 等前三条线推进之后，第四条线会更自然浮现

这个开放位可以作为未来的容器，留给下面这些可能性：

- 主动性/规划能力进一步增强
- 多 Agent 协同
- 个人知识库与工作流系统
- 更强的文件、浏览器、桌面操作能力
- 更强的个性化人格与长期陪伴体验

## 4. 当前阶段的优先级判断

建议优先级如下：

1. 记忆系统
2. Web Search
3. 自动任务
4. 未定义强化方向

背后的逻辑：

- 记忆定义“这个 Agent 能不能持续成为同一个 Agent”
- 搜索定义“这个 Agent 能不能接触真实世界”
- 自动任务定义“这个 Agent 能不能主动创造价值”

如果按系统依赖关系看，也基本是这个顺序。

## 5. 建议采用的阶段路线

### 5.1 近期两周执行重点

从 `2026-04-01` 开始，近期两周不再平均推进所有方向，而是按依赖关系推进：

1. 学 Claude Code 的上下文组装、Agent 编排、工具调用逻辑
2. 先把 Lecquy 自己的记忆系统地基打通
3. 再做 Claude Code 风格的 compact 原型和 cache-friendly 优化
4. RAG 只做 spike，不抢记忆主线

当前详细执行顺序已经下沉到 [`backend/20260408-4-记忆系统一期后端实施 开发规划.md`](./backend/20260408-4-记忆系统一期后端实施 开发规划.md)。

### Phase 1：从 AI Web 到 Personal Agent 底座

目标：

- 把“个人强 Agent”写成明确产品定位
- 重新整理系统中的记忆、会话、任务、产物边界
- 明确哪些模块是底座，哪些只是 UI 形态

重点：

- 统一 `.lecquy/` 下的长期状态定义
- 区分短期上下文、长期记忆、任务状态、文档产物
- 为后续记忆系统预留清晰数据结构

### Phase 2：记忆系统一期

目标：

- 先做可用、稳定、能解释的长期记忆

重点：

- 用户画像
- 项目/任务状态
- 会话摘要
- 检索与写回机制

### Phase 3：搜索系统一期

目标：

- 为 Agent 建立可靠的外部信息获取能力

重点：

- 搜索后端接入
- 搜索结果归一化
- 搜索摘要与引用
- 网页抓取与抽取

### Phase 4：自动任务一期

目标：

- 让 Agent 具备持续执行和跟进能力

重点：

- 任务调度
- 触发条件
- 执行记录
- 结果回传
- 用户确认机制

### Phase 5：高级强化

候选方向：

- 多模态记忆
- 图谱记忆
- 主动反思
- 复杂工作流自动化
- 多 Agent 协同

## 6. 做这个项目时应坚持的原则

- 不再把“聊天体验增强”当作唯一主线
- 优先建设 agent 基础设施，而不是只堆前端交互
- 优先解决长期价值，而不是短期 demo 效果
- 能本地化的能力尽量本地化
- 让记忆、搜索、任务三条线最终能彼此打通
- 每新增一个能力，都要问一句：它是否让 Lecquy 更像一个真正长期存在的个人 Agent？

## 7. 当前这版方向的简明总结

可以把这次重新定义概括为下面这段话：

> Lecquy 后续不再以“做一个 AI Web 聊天产品”为终点，而是以“做一个个人强 Agent 系统”为终点。现阶段的核心建设顺序是：先把记忆做强，再把搜索接进来，再让 Agent 真正开始自动执行长期任务。等这三条主线站稳后，再决定第四条新增方向。

## 8. 后续回看时可以先问自己的几个问题

- 这个需求是在强化聊天，还是在强化 Agent？
- 这个功能是否会增强长期记忆能力？
- 这个功能是否会增强接入现实世界的能力？
- 这个功能是否会增强主动执行能力？
- 这个功能是否会沉淀为系统能力，而不是一次性技巧？

如果以上问题大多回答为“否”，那它很可能不是当前阶段最重要的事情。
